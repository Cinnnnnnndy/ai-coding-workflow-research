# 后端代码 → 产品文档

这一步最容易被省略，但它是后续所有设计决策的基础。
代码里藏着产品的真实逻辑——业务规则、数据约束、实体关系——
设计师读懂它，才能做有根据的判断，而不是靠猜。

---

## 需求分析

### 用户与痛点

| 角色 | 典型场景 | 核心痛点 |
|------|---------|---------|
| 体验设计师 | 后端代码先于设计稿完成，需要快速理解「这个工具是做什么的」 | 看到一堆字段名和整数编码，不知道哪些和用户有关、哪些是内部机制；想做设计审查但没有业务上下文，审查结论流于表面 |
| 产品经理 | 接手无文档的存量工具，需要向新成员解释产品逻辑 | 只能靠运行软件来猜功能，边界不清，容易遗漏约束条件 |
| 前端开发 | 接手陌生代码仓，需要快速建立业务理解后再开发 | 字段命名用工程术语（`cmd_word: 0x30440100`），不知道这个值的含义；修改一处不知道会不会影响其他模块 |
| 新加入的团队成员 | 没有老员工传递背景，需要自学理解系统 | 文档散落在代码注释和口口相传中，找不到全貌 |

**设计师的深层痛点**：
工程工具类产品（固件配置、硬件调试、工业控制）的代码里充满了设计师不熟悉的技术概念——十六进制命令字、位字段、通信拓扑、表达式语法。这些不是障碍，是信息富矿。关键在于建立一套从「技术表达」翻译到「设计判断」的方法。

### 场景

1. **后端先行、无文档交付**：工程师完成了实现，但没有产品说明。设计师以代码为一手资料，反向建立对产品的理解，再做设计介入。
2. **新增视图的场景分析**：产品迭代时新增了若干界面，需要逐一分析：这个视图面向谁、用户在什么情境下打开它、解决什么问题、当前实现有哪些设计机会点。
3. **设计审查的上下文准备**：在跑 `/design-critique` 或 `/accessibility-review` 之前，先用此 skill 建立产品上下文，让后续审查结论有业务依据而非泛泛而谈。
4. **技术编码的设计语言翻译**：把工程师的表达（`pid_control_mode: 1`、`cmd_word: 0x30440100`）翻译成用户可理解的语言，建立设计-开发共同词汇表。
5. **存量系统设计改进**：改进现有工具的体验，需要先读懂它的数据结构和业务逻辑，才能判断哪些改动安全、哪些有风险。

### 设计点

- **翻译是设计工作**：把 `pid_control_mode: 1/2/3/4` 翻译成「慢升快降（省电优先）」，这个过程本身就是设计判断——选择什么语言、粒度多细、用什么隐喻。
- **产品文档决定审查质量**：带着「核心用户是在嘈杂机房里长时间使用的固件工程师」这个上下文去做审查，结论的精准度会高出 10 倍。
- **机会点分级决定优先级**：分清「前端独立实现」「需后端配合」「需产品层决策」，让改进建议可落地而不是空谈。

## 输入

| 类型 | 示例 | 说明 |
|------|------|------|
| 代码仓 | GitHub / GitCode 链接，或本地目录路径 | 主要读取 |
| 数据模型文档 | `data-model.md`、OpenAPI spec、ERD | 优先阅读，信息密度最高 |

| 新增视图说明 | 截图、Figma、功能描述 | 分析新视图时的补充输入 |

## 工作流

### Step 1 — 技术栈扫描（约 5 分钟）

读取：`package.json` / `requirements.txt` / `README` / `index.html`

确认：
- 语言与框架（原生 JS / Vue / React / Python...）
- 数据格式（YAML / JSON / XML / 二进制）
- 构建与部署方式（有无构建工具、是否支持 CDN 直接引入）
- 前端关心的核心依赖（图表库、YAML 解析器、Canvas 库、通信库）

> **openUBMC 案例**：`cooling-web-config` 是纯 HTML + CSS + JS，用 `js-yaml` CDN 解析 YAML，Canvas 绘制曲线编辑器，无需构建工具——这意味着所有工程级代码也必须保持同样的轻量约束。

### Step 1.5 — 功能模块识别与边界确认

扫描源码目录结构，按功能边界聚类，识别独立的功能模块。判断依据：
- 有独立的子目录或文件前缀命名
- 有自己的 WebView / Provider / 数据类型定义
- 使用场景完全不同，即便入口相邻或共享部分代码

区分「独立功能模块」与「同一模块的子功能」——子功能共享数据模型或操作上下文，归并到父模块里分析，不单独展开。

**输出模块清单，在继续之前发给用户确认**：

```
| 模块 | 一句话描述 | 是否独立分析 |
|------|-----------|------------|
| XXX  | ...       | ✅ 是      |
| YYY  | ...       | ❓ 待确认  |
```

如果遇到以下情况，**等用户确认再继续**：
- 两个功能共享部分代码但使用场景明显不同（入口相邻不等于是同一功能）
- 某个模块规模很小，不确定是否值得独立展开
- 某个功能被版本历史或注释标注为「已移除」「实验性」或「WIP」

确认之后，只对用户明确要分析的模块进入后续步骤，其余模块不展开。

---

### Step 2 — 数据模型分析（核心步骤）

读取：数据模型文档、样本数据文件、实体定义代码

**2a. 实体清单**
列出所有业务实体，每个记录：
- 实体名称（英文原名 + 中文业务含义）
- 核心字段（名称、类型、是否必填、取值范围）
- 枚举值映射：工程值 → 用户可读标签

```
示例：
pid_control_mode: 1 | 2 | 3 | 4
→ 1: 慢升快降（省电优先）
→ 2: 快升慢降（响应优先）
→ 3: 慢升慢降（稳定优先）
→ 4: 快升快降（激进模式）
```

**2b. 魔法值标注**
识别用特殊数字/字符串表示特殊状态的字段：

```
示例：
sys_hdds_max_temperature: 32768
→ 32768 = 「无效/获取不到」的内部标记
→ UI 必须转换为「—」或「不可用」，绝不能直接展示
```

**2c. 实体关系图**
哪些实体之间有引用关系？用层级列表表示：

```
CoolingConfig（全局）
└── CoolingArea（调速区域）
    ├── CoolingRequirement（温度需求，引用传感器）
    ├── CoolingPolicy（调速策略，含转速曲线）
    └── CoolingFan（绑定的风扇）
```

标注：哪些关联是用户必须手动建立的，哪些是自动推导的。

**2d. 动态字段标注**
标注有特殊语法或动态行为的字段：

```
示例：
sensor_reading: "<=/Scanner_Lm75_Inlet.Value"
→ CSR 动态语法：实时读取传感器值
→ UI 层不能把它当普通字符串处理
```

### Step 2+ — 陌生技术概念的识别与解释

读代码时，你会遇到各种「看不懂」的东西——一个奇怪格式的数字、一段自定义语法、一个专业术语缩写。这不是障碍，是信息密度最高的地方。

**判断一个技术概念是否需要解释的标准**：
> 如果一个刚入职的体验设计师/开发新员工看到这个词/值/格式，会不会有疑问？

会 → 需要在产品文档里解释。
不会 → 可以直接使用，不用额外说明。

---

#### 如何解释任何陌生技术概念：四步法

无论遇到什么陌生技术概念——特殊的数字格式、领域专用语法、硬件协议名称、缩写术语——都用这四个问题来解释清楚：

**① 它是什么？**（用一句话，不用行话）
把这个概念还原成最朴素的描述。不要用技术定义，用「这是一种……用来……的东西」的句式。

**② 为什么要这样设计？**（工程师的动机）
工程师不会无缘无故用一个奇怪的格式。找到他们这样做的原因——通常是为了节省空间、提高处理效率、或者遵循某个硬件规范。理解了动机，才能理解边界。

**③ 它具体是怎么工作的？**（用实例 + 图示说明）
只靠文字解释抽象概念很难被记住。用一个具体的真实例子走一遍，必要时画出结构图。「走一遍」比「讲原理」有效得多。

**④ 对用户/设计意味着什么？**（翻译回界面语言）
技术概念解释完了，最终要回答的问题是：这个概念在界面上应该怎么呈现？用户在哪一步会接触它？它对交互设计有什么约束或机会？

---

#### 什么时候用图示，什么时候用文字

| 情况 | 推荐方式 |
|------|---------|
| 概念有明确的结构（位分段、层级关系、网络拓扑） | 画结构图（ASCII 图或表格）|
| 概念涉及时序或流程（操作步骤、状态变化） | 画流程图或时序图 |
| 概念是一个类比（「就像……一样」） | 文字类比 + 一个对比例子 |
| 概念是枚举值（选项 A/B/C 各是什么意思） | 映射表格 |
| 概念是纯数值关系（公式、换算） | 带注释的计算示例 |

**原则：** 如果你读了自己写的解释，还是觉得模糊，就加一个图。图的目的不是美观，是消除歧义。

#### 三类高频出现的技术概念（附解释示范）

以下三类概念在工程工具类产品里极为常见。用它们作为「四步法」的练习示范。

---

**类型一：编码整数（一个数字里藏了多段信息）**

工程代码里常出现一个看起来像乱码的数字，比如 `0x30440100`。它实际上是把多段信息「压缩」进了一个整数。

*① 它是什么？*
一种把多个字段打包进同一个数字的存储方式。就像一个压缩文件——外面看是一个文件，解压后里面有很多东西。

*② 为什么这样设计？*
硬件通信时每次只能发送固定长度的数据（比如 4 个字节 = 32 位）。工程师把多个字段塞进这 32 位，既满足了协议要求，又一次性传递了所有必要信息。

*③ 它具体怎么工作？*（需要画图解释）

用十六进制写一个 32 位整数，正好是 8 个字符（`0x30440100`）。这不是巧合：

```
十六进制和二进制的对应关系：
每 1 位十六进制字符 = 4 位二进制（称为一个"nibble"）
8 位十六进制 × 4 = 32 位二进制

0x  3    0    4    4    0    1    0    0
   ↓    ↓    ↓    ↓    ↓    ↓    ↓    ↓
  0011 0000 0100 0100 0000 0001 0000 0000
  └─────────┘ └─────────┘ └─────────┘ ...
   第一个字段    第二个字段   第三个字段
```

工程师把字段边界设计在 nibble 边界上，这样用肉眼看十六进制字符就能分辨各字段——这是选择十六进制表示的根本原因。

字母的含义（十六进制用 0–9 和 A–F 表示 0–15）：
```
A=10  B=11  C=12  D=13  E=14  F=15
0xC = 12（十进制）= 1100（二进制）
```

同一个数里有时混用十六进制和十进制，原因是：有些字段代表「编码/地址」（用十六进制，因为位结构有意义），有些字段代表「数量/序号」（用十进制，因为数字本身有意义）。

用户如何获得这些字段定义？**不是从数字本身推导**，而是从外部规格文档（芯片手册、寄存器表、协议文档）查到的——文档里会写「第 24–31 位是功能码，第 16–23 位是命令码」，用户按照这个定义来填写或解读数字。

*④ 对设计意味着什么？*
- 用户不应该手动填写这个整数，UI 应该把各字段拆成独立输入框
- 用户填好各字段后，UI 在后台合并生成完整的编码值
- 需要显示「当前命令字预览」，让用户确认生成的值符合预期
- 字段之间可能有约束（某字段的值会影响另一字段的可选范围），需要联动校验

**类型二：领域表达式语言（产品自定义的迷你语法）**

代码里出现一种「不像普通代码也不像自然语言」的语法，比如 `ReadRegister(channel=2, offset=0x0C, length=4)` 或者 `<=/Scanner_Lm75_Inlet.Value`。这是产品自己定义的一套「迷你语言」。

*① 它是什么？*
一种专门为这个产品设计的指令语法，介于「人话」和「编程语言」之间。用于描述计算逻辑、查询条件或操作规则，不需要通用编程知识就能（理论上）看懂。

*② 为什么这样设计？*
让不会写代码的用户（运维人员、工程师）也能描述「我想做什么操作」，而不需要学习一门完整的编程语言。

*③ 它具体怎么工作？*（关键是区分不同的使用模式）

同一套表达式语法，往往有两种截然不同的使用方式：

```
调试模式（即席执行）          用例模式（脚本化复用）
─────────────────────         ──────────────────────
用户意图：「我现在想试         用户意图：「我要定义一套
试这条指令是否有效」           可反复执行的标准操作」

→ 填一条表达式                → 填多条表达式，组成序列
→ 立刻执行，看结果             → 保存为「用例」，按需触发
→ 不保存                      → 可共享给团队

类比：计算器 vs 宏录制
```

输入参数的作用：
```
不带参数：ReadRegister(channel=2, offset=0x0C)  ← 只能读这一个寄存器
带参数：  ReadRegister(channel=$ch, offset=$off) ← 模板，可复用

添加多个参数值（批量执行）：
ch = [0, 1, 2, 3]，offset = 0x0C
→ 自动执行 4 次，分别读取 channel 0/1/2/3 的同一个寄存器
→ 相当于「一个表达式，多组输入，对比结果」
```

*④ 对设计意味着什么？*
- 调试模式和用例模式是两种不同的使用意图，不能共用同一个表单加一个开关了事
- 切换模式时，用户已填写的内容不能丢失
- 参数输入需要类型提示和值域说明（`$ch` 应该是整数，取值 0–7）
- 批量执行的多组结果适合表格对比展示，不适合逐条显示

---

**类型三：通信拓扑（命令发给谁）**

代码里有各种设备地址（`0x48`、`0x2C`）和总线编号（`i2c_bus: 1`），还有「发送命令」「广播」之类的操作。这描述的是一个硬件通信网络。

*① 它是什么？*
多个硬件设备连接在一条共享线路上，每个设备有唯一「门牌号」（地址）。发送命令时需要指定门牌号，命令才能准确送达目标设备，不影响其他设备。

*② 为什么这样设计？*
硬件资源有限，不能为每对设备之间都拉一条专用线。共享总线 + 地址寻址，是一种经济实用的多设备通信方案。I2C、SPI、CAN 等都是这类协议，原理相同，细节不同。

*③ 它具体怎么工作？*

```
通信主机（发令者）
     │
     ├── 总线 0 ────┬── 地址 0x48：进风口温度传感器
     │              └── 地址 0x4A：排风口温度传感器
     │
     └── 总线 1 ────┬── 地址 0x2C：风扇控制器
                    └── 地址 0x58：电源管理芯片

发送命令 → 指定「总线 + 地址」→ 只有那个设备响应
广播命令 → 不指定地址（地址 0x00）→ 总线上所有设备都响应
```

命令字和通信拓扑的关系：命令字里通常有一个字段是「目标设备地址」，这个地址指向拓扑图里的某个节点。命令字决定「做什么」，拓扑决定「发给谁」。

*④ 对设计意味着什么？*
- 用户选择「发给谁」时，应该能看到设备列表或拓扑图，而不是手动填写地址数字
- 发送前应显示「将发送给：[设备名称]（地址 0x48，总线 0）」，让用户确认目标
- 广播操作影响所有设备，必须有二次确认警告
- 拓扑结构如果发生变化（新增设备），UI 应该能反映，不依赖用户手动更新地址

#### 遇到其他陌生技术概念时的处理原则

不是每个项目都有命令字或通信拓扑，但每个工程工具类产品都有它自己的「术语墙」。遇到任何看不懂的概念时：

1. **先问「这解决了什么问题」**——工程师不会无缘无故发明一个复杂概念，找到动机是理解的捷径。
2. **找最小可理解的例子**——不要试图先理解全貌，找一个最简单的真实值（如 `0x30440100`），走一遍完整的解析过程。
3. **用类比建立直觉**——「就像……」的类比往往比技术定义有效 10 倍。好类比的标准：听的人说「哦，就是那个意思啊」。
4. **画出来**——任何有结构的概念（层级、分段、网络），画出来之后通常立刻清晰。不需要漂亮，ASCII 图已经够用。
5. **最后问「所以呢」**——翻译回界面语言：这个概念对用户操作有什么影响？界面上应该怎么处理？

### Step 3 — 逐模块分析框架

对每个确认需要分析的功能模块，按以下 7 项固定结构逐一展开：

| 项次 | 内容 | 说明 |
|------|------|------|
| 1 | **需求背景** | 这个功能从哪来？为什么存在？在整体技术栈或产品体系中的位置 |
| 2 | **概念解释 + 图示** | 核心技术概念用最简语言解释，搭配结构图（不懂业务的人也能看懂） |
| 3 | **使用场景** | 2–4 个具体场景，用叙事描述：用户在做什么 → 遇到什么 → 怎么用 → 得到什么 |
| 4 | **目标用户** | 角色、使用频率、使用环境、核心诉求、典型触发点 |
| 5 | **痛点分析** | 当前没有工具时的操作成本（量化）；每个痛点用一句话提炼核心 |
| 6 | **设计机会点** | 🟢 前端可独立实现 / 🟡 需后端配合 / 🔴 需产品层决策 |
| 7 | **竞品分析 + 专利潜力** | 见下方格式要求 |

**第 7 项：竞品分析 + 专利潜力评估**

竞品分析的重点是找「差距」，而不是罗列功能：

```
| 竞品 / 行业方案 | 具备的能力 | 与本功能的差距 |
|----------------|-----------|--------------|
```

> 差距是设计决策的依据——「我们能做而别人没做」才是机会，「别人做了一样的」只是参考。

专利潜力评估回答三个问题：
1. 这个功能组合（数据结构 × 操作方式 × 应用场景）在行业内是否有先例？
2. 差异化点是哪几个关键词的组合（注意：是**组合**，而非单个关键词）？
3. 专利潜力评级：**高**（组合未见先例）/ **中**（组合新颖但单项不稀缺）/ **低**（已有公开先例）

读取：解析逻辑、校验规则、导出逻辑、操作顺序相关代码

还原用户完成一个完整任务的操作顺序，标注：

- 可任意顺序完成的步骤

### Step 5 — 设计机会点汇总（三列流视图）

将所有模块的设计机会点汇总为一张**三列流视图**，表达「痛点 → 洞察 → 策略」的因果链，而不是平铺的机会点列表。

**三列结构**

| 列 | 内容 | 写法要求 |
|----|------|---------|
| 第一列 | **用户痛点** | 具体、可量化，带操作成本（来自各模块的痛点分析） |
| 第二列 | **洞察汇聚** | 从多个痛点中提炼的深层原因，通常比痛点数量少（多对一） |
| 第三列 | **设计策略** | 对应洞察的具体设计方向，标注 🟢/🟡/🔴 实现层级 |

三列之间用连线标注「哪些痛点汇聚成哪条洞察，哪条洞察导出哪个策略」。

**洞察汇聚的写法**：不是痛点的复述，而是提炼「为什么这些痛点会同时出现」的深层原因。

```
错误写法：「命令字操作难」（只是痛点 P1 的重复）
正确写法：「翻译比填写更高频」（解释了 P1+P2 为什么同时出现的根因）
```

在 Step 8 的汇总幻灯片里，这张三列流视图以 HTML 可视化渲染（连线 + 色块），不是表格。

### Step 8 — 生成 16:9 场景分析 HTML 材料

分析完成后，将每个功能场景的研究结论输出为一份可独立传阅的 HTML 幻灯片文件（`[product-name]-slides.html`），格式类似 PPT，每页 16:9 比例，供设计评审、团队对齐、向上汇报使用。

**整体结构**

- 首页：产品名称 + 工具范围 + 阅读对象说明
- 每个功能模块占一组幻灯片，内部按以下固定页顺序：

| 页次 | 内容 | 说明 |
|------|------|------|
| 1 | **需求背景** | 这个功能从哪来？为什么存在？在整体技术栈或产品体系中的位置 |
| 2 | **概念解释 + 图示** | 核心技术概念用最简语言解释，搭配 ASCII 图、对照表、结构示意图（不懂业务的人也能看懂） |
| 3 | **使用场景** | 2–4 个具体场景，用叙事描述：用户在做什么任务 → 遇到什么 → 怎么用这个功能 → 得到什么 |
| 4 | **目标用户** | 用户类型表格：角色、使用频率、核心诉求、典型触发点 |
| 5 | **痛点分析** | 每个痛点一段，说明当前没有工具时的操作成本，用「痛点核心：」一句话提炼 |
| 6 | **设计机会点** | 按 🔴 高 / 🟡 中 / 🟢 低优先级分级，每条机会点说明当前问题 + 建议方向 |
| 7 | **竞品分析 + 专利潜力** | 竞品差距表（重点是差距而非功能对比）+ 专利潜力评级（高/中/低）+ 差异化关键词组合 |
| 汇总页 | **三列流视图** | 跨所有模块汇总：第一列痛点 → 第二列洞察汇聚 → 第三列设计策略，三列之间用连线表达因果链 |

**幻灯片技术规格**

```
比例：16:9（宽高比 16/9，推荐容器尺寸 1280×720px）
主题：深色背景（#0d1117 或接近），配色参考产品本身色系
字体：系统 sans-serif 栈，正文 ≥ 14px，标题 24–36px
每页内容量：避免满屏文字；重点信息用大字、高亮色块、图表呈现
导航：← → 键盘翻页 + 左右点击按钮 + 幻灯片编号（当前 / 总数）
代码块：等宽字体，语法着色（可用简单 class 区分）
```

**内容写作标准**

- 面向「不熟悉这块业务的体验设计师/开发新员工」：不预设领域知识，概念页要真正讲清楚
- 场景描述用完整叙事，而非列点：「工程师正在排查……他打开……他看到……」
- 痛点用具体数字量化（「每次迭代需要 5–10 分钟」「平均要迭代 5–10 次」）
- 设计机会点区分「前端可独立实现」「需后端配合」「需产品层决策」

**参考示例**

`tool-views-analysis.md` 文档是这类分析内容的文字原型，HTML 材料是把同样的内容转换为可视化幻灯片格式。每页信息量对应文档中的一个小节，而不是把整节文字直接搬上去——要提炼、可视化、加图示。

## 来自 openUBMC Studio 的实践笔记

**先读数据模型，再读代码。** `data-model.md` 里的信息密度比源代码高得多——它已经做了一次结构化，你从这里开始能节省大量时间。

**样本数据是最好的字段说明书。** `sample-cooling.yaml` 里的真实值告诉你字段的实际值域，比类型定义（`string`）更有用。

**魔法值是最优先要找的。** `32768`、`-1`、`"N/A"` 这类特殊值往往隐藏在业务逻辑里，不找就会漏到 UI 上——这是最容易被用户骂的那类 bug。

**十六进制不是障碍，是结构信号。** 看到 `0x30440100` 不要跳过——这个写法告诉你这个值被分成了 4 段，每段 8 位，每段有独立含义。设计师的工作是把这些「段」翻译成用户能填写和理解的字段。

**位字段的来源永远是外部文档，不是命令码本身。** 用户不是「拿到命令码来解析字段」，而是「先有规格文档知道各段含义，再按规格填写命令码」。UI 的目标是让规格文档里的字段定义变成用户的操作界面，而不是让用户手动做位运算。

**调试模式和用例模式的根本差异是使用意图，不是功能差异。** 调试模式是「我要试试这条命令」，用例模式是「我要定义一套可重复的操作」。两者的 UI 结构应该反映这种意图差异，而不是共用同一个表单加一个 toggle。

**通信拓扑的可见性直接影响操作安全性。** 工程师发一条命令影响的是一个具体的物理设备，如果 UI 里看不出来命令发给了谁，就容易误操作影响不相关的设备。让「命令目标」在每次操作前都可见，是工程工具类产品最基本的安全设计。

**产品文档是审查工具的放大镜。** 同样的「对比度不足」，有了「核心用户是在嘈杂机房里长时间使用的固件工程师」这个上下文，严重程度和修复优先级会完全不同。
