# 后端代码 → 产品文档

这一步最容易被省略，但它是后续所有设计决策的基础。
代码里藏着产品的真实逻辑——业务规则、数据约束、实体关系——
设计师读懂它，才能做有根据的判断，而不是靠猜。

---

## 需求分析

### 用户与痛点

| 角色 | 典型场景 | 核心痛点 |
|------|---------|---------|
| 体验设计师 | 后端代码先于设计稿完成，需要快速理解「这个工具是做什么的」 | 看到一堆字段名和整数编码，不知道哪些和用户有关、哪些是内部机制；想做设计审查但没有业务上下文，审查结论流于表面 |
| 产品经理 | 接手无文档的存量工具，需要向新成员解释产品逻辑 | 只能靠运行软件来猜功能，边界不清，容易遗漏约束条件 |
| 前端开发 | 接手陌生代码仓，需要快速建立业务理解后再开发 | 字段命名用工程术语（`cmd_word: 0x30440100`），不知道这个值的含义；修改一处不知道会不会影响其他模块 |
| 新加入的团队成员 | 没有老员工传递背景，需要自学理解系统 | 文档散落在代码注释和口口相传中，找不到全貌 |

**设计师的深层痛点**：
工程工具类产品（固件配置、硬件调试、工业控制）的代码里充满了设计师不熟悉的技术概念——十六进制命令字、位字段、通信拓扑、表达式语法。这些不是障碍，是信息富矿。关键在于建立一套从「技术表达」翻译到「设计判断」的方法。

### 场景

1. **后端先行、无文档交付**：工程师完成了实现，但没有产品说明。设计师以代码为一手资料，反向建立对产品的理解，再做设计介入。
2. **新增视图的场景分析**：产品迭代时新增了若干界面，需要逐一分析：这个视图面向谁、用户在什么情境下打开它、解决什么问题、当前实现有哪些设计机会点。
3. **设计审查的上下文准备**：在跑 `/design-critique` 或 `/accessibility-review` 之前，先用此 skill 建立产品上下文，让后续审查结论有业务依据而非泛泛而谈。
4. **技术编码的设计语言翻译**：把工程师的表达（`pid_control_mode: 1`、`cmd_word: 0x30440100`）翻译成用户可理解的语言，建立设计-开发共同词汇表。
5. **存量系统设计改进**：改进现有工具的体验，需要先读懂它的数据结构和业务逻辑，才能判断哪些改动安全、哪些有风险。

### 设计点

- **翻译是设计工作**：把 `pid_control_mode: 1/2/3/4` 翻译成「慢升快降（省电优先）」，这个过程本身就是设计判断——选择什么语言、粒度多细、用什么隐喻。
- **产品文档决定审查质量**：带着「核心用户是在嘈杂机房里长时间使用的固件工程师」这个上下文去做审查，结论的精准度会高出 10 倍。
- **机会点分级决定优先级**：分清「前端独立实现」「需后端配合」「需产品层决策」，让改进建议可落地而不是空谈。

## 输入

| 类型 | 示例 | 说明 |
|------|------|------|
| 代码仓 | GitHub / GitCode 链接，或本地目录路径 | 主要读取 |
| 数据模型文档 | `data-model.md`、OpenAPI spec、ERD | 优先阅读，信息密度最高 |
| 样本数据 | `sample-*.yaml`、`*.json` 样本文件 | 快速理解字段实际值域 |

| 新增视图说明 | 截图、Figma、功能描述 | 分析新视图时的补充输入 |

## 工作流

### Step 1 — 技术栈扫描（约 5 分钟）

读取：`package.json` / `requirements.txt` / `README` / `index.html`

确认：
- 语言与框架（原生 JS / Vue / React / Python...）
- 数据格式（YAML / JSON / XML / 二进制）
- 构建与部署方式（有无构建工具、是否支持 CDN 直接引入）
- 前端关心的核心依赖（图表库、YAML 解析器、Canvas 库、通信库）

> **openUBMC 案例**：`cooling-web-config` 是纯 HTML + CSS + JS，用 `js-yaml` CDN 解析 YAML，Canvas 绘制曲线编辑器，无需构建工具——这意味着所有工程级代码也必须保持同样的轻量约束。

### Step 2 — 数据模型分析（核心步骤）

读取：数据模型文档、样本数据文件、实体定义代码

**2a. 实体清单**
列出所有业务实体，每个记录：
- 实体名称（英文原名 + 中文业务含义）
- 核心字段（名称、类型、是否必填、取值范围）
- 枚举值映射：工程值 → 用户可读标签

```
示例：
pid_control_mode: 1 | 2 | 3 | 4
→ 1: 慢升快降（省电优先）
→ 2: 快升慢降（响应优先）
→ 3: 慢升慢降（稳定优先）
→ 4: 快升快降（激进模式）
```

**2b. 魔法值标注**
识别用特殊数字/字符串表示特殊状态的字段：

```
示例：
sys_hdds_max_temperature: 32768
→ 32768 = 「无效/获取不到」的内部标记
→ UI 必须转换为「—」或「不可用」，绝不能直接展示
```

**2c. 实体关系图**
哪些实体之间有引用关系？用层级列表表示：

```
CoolingConfig（全局）
└── CoolingArea（调速区域）
    ├── CoolingRequirement（温度需求，引用传感器）
    ├── CoolingPolicy（调速策略，含转速曲线）
    └── CoolingFan（绑定的风扇）
```

标注：哪些关联是用户必须手动建立的，哪些是自动推导的。

**2d. 动态字段标注**
标注有特殊语法或动态行为的字段：

```
示例：
sensor_reading: "<=/Scanner_Lm75_Inlet.Value"
→ CSR 动态语法：实时读取传感器值
→ UI 层不能把它当普通字符串处理
```

### Step 2+ — 特殊技术数据格式的识别与翻译

工程工具类产品常出现三类设计师不熟悉但必须读懂的技术数据格式。遇到时用以下框架分析。

---

#### 模式 A：位字段编码整数（Bit-field Encoded Integer）

**是什么**：用一个整数同时编码多段信息，不同「位」区间代表不同的字段。常见于硬件通信协议、固件命令接口、寄存器操作。

**为什么用十六进制**：
一个 32 位整数用十进制写是一串没有规律的数字（如 `805830912`），肉眼无法感知结构。用十六进制写则恰好是 8 位（如 `0x30440100`）：

```
0x  3  0  4  4  0  1  0  0
    ↑  ↑  ↑  ↑  ↑  ↑  ↑  ↑
   每位十六进制字符 = 4 个二进制位（一个 nibble）
   8 位 hex × 4 bits = 32 bits，正好覆盖整个整数
```

因此，当字段边界和 nibble（4 位）对齐时，工程师靠肉眼就能读出各段含义。这是工程师选择十六进制的根本原因：**十六进制是 32 位整数的「天然分隔符」**。

**0x 前缀中的字母含义**：
十六进制用 0–9 和 A–F 表示 0–15，字母即超过 9 的值：
```
0xC = 12（十进制）= 1100（二进制）
      意味着该 nibble 的高两位都是 1，低两位都是 0
```

**为什么同一个命令字里混用十六进制和十进制**：
- **十六进制**：用于表示「位模式有意义」的字段，如功能码、命令码、设备地址——这些值的二进制位结构本身携带信息，换用十进制会丢失对齐感。
- **十进制**：用于表示「数量/索引有意义」的字段，如通道编号（第 3 路）、寄存器偏移（第 12 个字节）——这些值对用户有直接的数数意义，用十六进制反而增加认知负担。
- 同一命令字里两者混用，是因为这个命令字里有些字段是「编码」、有些字段是「数量」，它们遵循不同的认知逻辑。

**如何分析位字段命令字**：

1. **确认总位宽**：是 8-bit、16-bit 还是 32-bit？
2. **找字段划分规范**：通常来自硬件规格文档（datasheet）、寄存器手册（register map）或 FPGA 规格，而非从命令码本身解析出来——工程师是先有规范、再按规范填写命令字。
3. **列出每个字段**：

```
示例（32-bit 命令字 0x30440100）：
Bits 31–24: 0x30 → 命令码（Command Code）= 功能大类
Bits 23–16: 0x44 → 功能码（Function Code）= 具体操作
Bits 15–8:  0x01 → 参数区高字节
Bits 7–0:   0x00 → 参数区低字节
```

4. **找出「功能码与命令码的重叠关系」**：在某些协议中，命令码的某几位本身就编码了功能码的大类，形成层级结构（如命令码高 4 位 = 功能大类，低 4 位 = 子功能）。有重叠说明这两个字段不是独立的，改一个会影响另一个。
5. **标注「位编号」**：位编号（bit number）是该位在整数中的位置，通常从 0（最低位）数起。Bit 0 = 最右边一位，Bit 31 = 最左边一位。

**UI 设计含义**：
- 用户输入一个完整命令字 → UI 应该能「解析展示」各字段（而非让用户手动计算）
- 用户填写各字段 → UI 应该能「合并生成」最终命令字
- 位编号和偏移量应来自用户手头的规格文档，UI 需要提供参照——用户不是从命令字本身推导这些值，而是从外部文档查到后填进来。

**用户痛点**：
- 拿到一个命令字（如 `0x30440100`）不知道各段含义，需要反复查规格文档
- 手动计算各字段值域容易出错，没有即时校验
- 字段之间有约束关系（改命令码可能使功能码失效），界面上看不出来

#### 模式 B：领域表达式语言（Domain Expression Language）

**是什么**：产品自定义的一种表达式/公式语法，用于描述计算逻辑、查询条件或操作规则，而不是通用编程语言。

**典型结构**：
```
表达式 = 操作符 + 操作数 + 参数
示例：ReadRegister(channel=2, offset=0x0C, length=4)
```

**调试模式 vs 用例模式**：
- **调试模式（Debug Mode）**：表达式直接执行，结果立刻返回。用于工程师测试单条指令是否工作。用户是「我要验证这个命令」。
- **用例模式（Use Case Mode）**：把多条表达式组合成一个可重复执行的测试用例或操作序列。用户是「我要定义一套标准操作流程」。
- 两种模式本质上是「即席执行」vs「脚本化复用」，是完全不同的使用意图，UI 结构应该区分清楚。

**输入参数的目的**：
表达式里的参数（如 `channel`、`offset`）允许用户在不改变表达式结构的情况下，用不同的值运行同一逻辑。目的是：
- **复用**：同一个表达式模板，换个 channel 值就能用在不同设备上
- **批量测试**：用多组参数值跑同一表达式，对比结果

**添加多个参数值的作用**：
为一个参数指定多个值（如 `channel = [0, 1, 2, 3]`），表示对每个值都执行一次表达式——这是批量验证的简化方式，不需要手动重复执行 N 次。相当于一个轻量级的参数化测试。

**用户痛点**：
- 表达式语法需要记忆，没有自动补全或语法提示
- 调试模式和用例模式之间切换时，上下文（已填写的参数）经常丢失
- 多参数值的结果没有对比视图，只能逐条查看
- 表达式出错时提示信息是工程师视角（「语法错误第 3 行」），而不是用户视角（「channel 参数需要是整数」）

**UI 设计含义**：
- 模式切换需要持久化用户已输入的内容
- 参数输入需要类型校验和值域提示（尤其是 offset 这类来自外部规格的值）
- 多参数值的执行结果适合表格对比展示
- 语法错误提示需要定位到具体参数，而不是报告行号

#### 模式 C：硬件通信拓扑中的命令对象（Hardware Communication Topology）

**是什么**：产品涉及向硬件设备发送命令，这些设备通过某种通信总线（如 I2C、SPI、UART）组织成一个网络（拓扑）。命令不是「广播」给所有人，而是精确发送给特定对象。

**I2C 拓扑与命令字的关系**（以 I2C 为例，原理可类推到其他总线协议）：
- I2C 是一种两线制通信协议，一条数据线（SDA）、一条时钟线（SCL），多个设备共享同一总线
- 每个设备有唯一的**地址**（7-bit 或 10-bit），命令通过地址路由到目标设备
- 命令字通常指定「要操作哪个设备」「对该设备的哪个寄存器」「执行什么操作（读/写）」
- 命令字和 I2C 拓扑的关系：命令字里的某个字段（如设备地址位）决定了这条命令发给拓扑里的哪个节点

**命令字发送给哪些对象**：
- **直接目标**：特定 I2C 地址对应的物理设备（如温度传感器、风扇控制芯片、电源管理 IC）
- **通过代理**：有时命令先发给主控（BMC / FPGA），主控再转发给下游设备
- **广播地址**：I2C 有保留的广播地址（0x00），命令会发给总线上所有设备——需要在 UI 上明确标注这是危险操作

**如何在产品文档中表示拓扑**：
```
通信主机（BMC）
├── I2C Bus 0
│   ├── 0x48 — 进风口温度传感器（LM75）
│   └── 0x4A — 排风口温度传感器（LM75）
└── I2C Bus 1
    ├── 0x2C — 风扇控制器（MAX31790）
    └── 0x58 — 电源管理（PMBus）
```

**用户痛点**：
- 拓扑图不可见：用户不知道当前命令会影响哪些设备，操作一个字段可能意外影响另一路总线
- 设备地址需要查规格文档，没有下拉选择，容易填错
- 拓扑变化（新增设备）需要手动更新命令字里的地址字段，没有联动

**UI 设计含义**：
- 对命令对象提供可视化选择（设备列表/拓扑图），而不是让用户手动填写地址
- 命令执行前显示「将发送给：[设备名称] @ [地址]」，确认目标
- 广播地址需要二次确认警告

### Step 3 — 新增视图的分析框架

当产品新增了若干界面时，对每个视图逐一分析以下四个维度，输出为结构化文档：

**3a. 场景**
这个视图在什么情境下被打开？用完整的叙事描述：
```
示例：
工程师正在排查一台服务器的散热异常。他需要向某个风扇控制芯片
发送一条读寄存器命令，确认当前的 PWM 占空比是否符合配置。
他打开「命令字构造」视图，按照芯片手册的字段定义，逐段填写
命令码、功能码和参数，最后点击发送并查看返回值。
```
场景要详细到：用户在做什么任务→遇到什么问题→打开这个视图→执行什么操作→得到什么结果。

**3b. 用户**
谁在用这个视图？不只是职业标签，还要说明：
- 他们的技术背景（会不会读十六进制？熟悉硬件协议吗？）
- 使用频率（每天用还是偶尔排查时才用？）
- 使用环境（在实验室？在嘈杂机房？在远程桌面里？）
- 他们最在意的成功指标（发送成功率？排查速度？操作记录可追溯？）

**3c. 解决的问题**
如果没有这个视图，用户怎么做这件事？当前方案的痛点是什么？
```
如果没有命令字构造视图，工程师需要：
1. 手动按位计算命令字（容易出错）
2. 用命令行工具发送原始十六进制字符串（没有校验）
3. 查阅两份文档（命令字规格 + 设备地址表）才能组出一条完整命令
→ 平均每次排查需要 15–20 分钟，且容易因填错地址影响其他设备
```

**3d. 设计机会点**
基于场景和痛点，列出可以在设计层面改善的方向：
- 🟢 前端可独立实现
- 🟡 需后端配合
- 🔴 需要产品层决策

---

### Step 4 — 业务流程还原

读取：解析逻辑、校验规则、导出逻辑、操作顺序相关代码

还原用户完成一个完整任务的操作顺序，标注：
- 必须按顺序完成的步骤（先定义实体，再建立引用）
- 可任意顺序完成的步骤
- 步骤间的依赖约束

### Step 5 — 设计机会点汇总

对照以下维度逐一检查，分三级标注：

**🟢 前端可独立实现**（不改后端、优先推进）
**🟡 需后端配合**（增加字段、调整数据结构）
**🔴 需要产品层决策**（涉及功能边界、需 PM 对齐）

| 维度 | 检查项 |
|------|--------|
| **信息架构** | 层级是否清晰？用户在哪里最容易迷失？分组是否符合用户心智模型？ |
| **引导性** | 新用户如何知道从哪里开始？必须按顺序的步骤是否有可见线索？ |
| **状态反馈** | 错误、成功、等待状态是否有可见反馈？错误提示是否描述了「什么错了 + 怎么修」？ |
| **依赖可见性** | 实体间的绑定关系是否可见？命令发给哪个设备是否明确？ |
| **数据量扩展性** | 10 条数据和 100 条数据体验是否一样？ |
| **技术术语暴露** | 工程字段名是否直接展示给用户？枚举整数是否未经转换直接显示？十六进制命令字是否被解析为人类可读形式？ |
| **操作安全性** | 危险操作（广播命令、覆盖配置）是否有二次确认？操作是否可撤销？ |

## 来自 openUBMC Studio 的实践笔记

**先读数据模型，再读代码。** `data-model.md` 里的信息密度比源代码高得多——它已经做了一次结构化，你从这里开始能节省大量时间。

**样本数据是最好的字段说明书。** `sample-cooling.yaml` 里的真实值告诉你字段的实际值域，比类型定义（`string`）更有用。

**魔法值是最优先要找的。** `32768`、`-1`、`"N/A"` 这类特殊值往往隐藏在业务逻辑里，不找就会漏到 UI 上——这是最容易被用户骂的那类 bug。

**十六进制不是障碍，是结构信号。** 看到 `0x30440100` 不要跳过——这个写法告诉你这个值被分成了 4 段，每段 8 位，每段有独立含义。设计师的工作是把这些「段」翻译成用户能填写和理解的字段。

**位字段的来源永远是外部文档，不是命令码本身。** 用户不是「拿到命令码来解析字段」，而是「先有规格文档知道各段含义，再按规格填写命令码」。UI 的目标是让规格文档里的字段定义变成用户的操作界面，而不是让用户手动做位运算。

**调试模式和用例模式的根本差异是使用意图，不是功能差异。** 调试模式是「我要试试这条命令」，用例模式是「我要定义一套可重复的操作」。两者的 UI 结构应该反映这种意图差异，而不是共用同一个表单加一个 toggle。

**通信拓扑的可见性直接影响操作安全性。** 工程师发一条命令影响的是一个具体的物理设备，如果 UI 里看不出来命令发给了谁，就容易误操作影响不相关的设备。让「命令目标」在每次操作前都可见，是工程工具类产品最基本的安全设计。

**产品文档是审查工具的放大镜。** 同样的「对比度不足」，有了「核心用户是在嘈杂机房里长时间使用的固件工程师」这个上下文，严重程度和修复优先级会完全不同。
