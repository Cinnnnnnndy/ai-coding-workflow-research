# 后端代码 → 设计反向分析 · 工程级代码生成

本 skill 对应 openUBMC Studio L1 协作中「后端先行」场景：
后端交付代码或配置后，设计师不是从零出发，而是以代码为锚点，
反向建立设计规范、发现问题、生成可直接接入工程的前端代码。

---

## 实际走过的完整协同流程

本 skill 来自 openUBMC 蓝区一次完整的「后端先行」协同实践，以下是实际经历的五步：

| 阶段 | 输入 | 工具/方法 | 产出 |
|------|------|----------|------|
| **Step 1** 接收后端交付物 | 代码仓：`cooling-web-config`、`data-model.md`、`sample-cooling.yaml`、`cooling-excel-parser` | 阅读 + 梳理 | 理解产品是什么、做什么 |
| **Step 2** 读代码 → 产品文档 | 以上所有后端交付物 | `/code-to-product-doc`（独立 skill） | 功能定义、用户场景、数据结构、核心流程、设计机会点、注意事项 |
| **Step 3** 产品文档 → 设计审查 | Step 2 的产品文档作为上下文 | `/design-critique` + `/accessibility-review` | 分级问题清单（🔴 Critical / 🟡 Moderate / 🟢 Minor）+ WCAG 2.1 AA 对照报告 |
| **Step 4** 审查结论 → 工程实现 | 审查报告 + 后端数据结构 | Edit/Write 组件代码 | 吸顶导航、可收起面板、步骤引导、静态 HTML 工程级代码，部署到 GitHub Pages |
| **Step 5** 验证与迭代 | 截图反馈 | 排查 → 修复 → 再验证 | 可用的线上 Demo |

> Step 2（代码 → 产品文档）是最容易被省略、但最不该省略的一步。它是后续所有设计决策的基础，也是设计师和工程师建立共同语言的节点。该步骤已提炼为独立 skill：`/code-to-product-doc`。

## 需求分析

### 用户

| 角色 | 使用场景 | 关注点 |
|------|---------|--------|
| 体验设计师 | 收到后端代码后介入，做设计分析和质量检查 | 能读懂代码里的业务含义；发现问题有据可依 |
| 前端开发工程师 | 接收设计侧输出的工程代码，直接集成 | 代码和后端数据结构严格对齐；不引入新的技术债 |
| 后端工程师 | 查看设计侧的可访问性和设计问题反馈 | 问题描述精准，能明确哪些需要后端配合改 |
| 项目负责人 | 评估设计介入后的改进方向和优先级 | 机会点有明确的难度和收益评估 |

### 场景

1. **后端先行交付**：后端代码或配置先于设计稿完成，设计师以代码为一手资料，反向建立对产品的理解。
2. **工程工具设计**：面向专业用户的配置工具、管理后台，没有消费端那样的视觉设计规范，但有更严格的可访问性和效率要求。
3. **设计质量审查**：开发已有现成实现，设计师作为质量守门员介入，找问题、提改进方向，而非全部推翻重做。
4. **设计-开发共同语言建立**：设计师和后端工程师通过代码中的字段命名、数据结构建立共同词汇表，减少沟通摩擦。

### 解决的问题

- **设计师看不懂代码**：代码里有大量对设计工作有价值的信息（枚举值、魔法值、实体关系），但设计师缺少读取和整理这些信息的方法。第 1 步「读懂后端交付物」提供了系统的拆解方式。
- **可访问性问题在后期才暴露**：字号 11px、颜色对比不足、Canvas 无键盘支持——这些问题如果在代码层面早发现，比产品上线后再改成本低得多。
- **设计建议缺乏工程依据**：「这里不好看」和「这里 font-size: 11px 低于 WCAG 2.1 AA 建议的 14px 等效最小值」，后者更容易被工程团队接受和排期。
- **设计改动不知道影响范围**：哪些改动设计师自己能做（前端独立实现），哪些需要后端配合，哪些需要产品层决策——机会点分析把这三类明确分开。

### 设计点

- **从代码提取业务语言**：`pid_control_mode: 1/2/3/4` 本身毫无意义，设计师的工作是将它翻译成用户能理解的「慢升快降（省电优先）」——这个翻译过程就是设计价值。
- **魔法值是 UI 层的责任**：`32768` 是后端内部约定，UI 绝不能直接展示。设计分析的目标之一就是发现这类「后端约定泄漏到用户界面」的问题。
- **可访问性先于视觉设计**：对工程工具类界面，可访问性不是锦上添花，是基础要求——专业用户长时间使用，字号和对比度的问题会在每天 8 小时里持续伤害效率。
- **分层推进，不全推翻**：区分「前端独立实现」「需后端配合」「需产品决策」三类机会点，让改进建议可落地，而不是理想化的全部重做。

## 工作流

### 第 1 步 — 读懂后端交付物

接收并解析后端交付的所有内容，写入 `code-reading.md`：

**1a. 识别交付物类型**
- 配置文件（YAML / JSON / TOML）：提取字段结构、枚举值、约束条件、魔法默认值
- 数据模型文档（Markdown / OpenAPI）：提取实体关系、字段类型、必填项、校验规则
- 源代码（HTML + CSS + JS）：提取组件结构、状态管理、事件处理，注意字号/颜色硬编码
- VSCode 插件或 CLI 工具：理解用户的实际工作上下文（在哪个工具里操作）

**1b. 建立数据字典**
对每个关键字段记录：
- 字段名（原始命名）与数据类型
- 业务含义（用操作人员能理解的语言，而非工程术语）
- 取值范围与枚举映射（如 `pid_control_mode: 1/2/3/4` → 「慢升快降/快升慢降/慢升慢降/快升快降」）
- 魔法默认值标注（如 `32768` 表示「无效/不可用」，对用户不友好，需在 UI 层转换）
- 与界面元素的对应关系

**真实案例参考 — cooling-web-config 字段梳理：**
```
smart_cooling_mode: "EnergySaving" | "LowNoise" | "HighPerformance" | "Custom" | "LiquidCooling"
  → UI 应展示：节能 / 低噪 / 高性能 / 自定义 / 液冷
  → 切换模式会联动影响：target_temperature_celsius 的有效范围

pid_control_mode: 1 | 2 | 3 | 4
  → UI 应展示带图解的选项：慢升快降（省电优先）/ 快升慢降（响应优先）/ ...
  → 整数值对用户完全无意义，必须在 UI 层做映射

sys_hdds_max_temperature: 32768  ← 魔法值
  → 32768 是「无效/获取不到」的内部标记，UI 应显示「—」或「不可用」
  → 绝对不能把 32768 直接展示给用户
```

**1c. 梳理实体关系**
- 哪些实体之间有 1:N 关系（如 `CoolingArea` 引用 `CoolingRequirement` + `CoolingPolicy` + `CoolingFan`）
- 哪些 ID 是「本地标识」vs「全局唯一 ID」（影响引用方式）
- 哪些字段使用 CSR 动态语法（如 `<=/Scanner_Lm75_Inlet.Value`），UI 层需要特殊处理

若有不明确的字段含义或业务逻辑，**主动向用户确认**，不要猜测。

### 第 2 步 — 可访问性审查

基于对代码/配置的理解，写入 `accessibility-audit.md`。
重点关注工程工具类界面（不同于消费者 app，此类工具的用户是专业人员，但更长时间使用）。

**审查维度（对照 WCAG 2.1 AA）：**

| 维度 | 检查项 |
|------|--------|
| 可感知性 | 色彩对比度是否 ≥ 4.5:1（正文）/ 3:1（大字体和 UI 组件）；错误状态是否仅依赖颜色区分 |
| 可操作性 | 所有功能是否可通过键盘完成；Canvas 类组件（如曲线编辑器）是否提供键盘 / 表格替代操作 |
| 可理解性 | 错误提示是否描述了「什么错了」和「如何修复」；技术字段名是否有人类可读标签 |
| 鲁棒性 | 是否使用语义化 HTML；ARIA 属性是否正确；表单 label 是否与 input 关联 |

**真实案例发现 — cooling-web-config 可访问性问题：**
```
❌ font-size: 12px 用于表单 label（低于 WCAG 建议的 14px 等效最小值）
❌ font-size: 11px 用于字段错误消息（几乎不可读）
❌ .field-error 仅用红色边框 + 背景区分，无图标，对色觉障碍用户不友好
❌ Canvas 曲线编辑器（curve-editor.js）无键盘操作支持，仅靠鼠标拖拽
❌ 错误列表 #error-list 没有 role="alert" 或 aria-live，屏幕阅读器无法感知
⚠️ cursor: crosshair 用于曲线画布 — 暗示「绘制」而非「数据编辑」，语义不准确
```

**输出格式：**
```markdown
## 问题清单
| # | 维度 | 位置/组件 | 问题描述 | 严重程度 | 受影响用户 | 修复建议 | 需后端配合 |
|---|------|----------|---------|---------|-----------|---------|----------|

## 已符合项
[记录已做好的可访问性实践，避免后续迭代破坏]

## 待后端配合项
[需要后端增加字段（如人类可读枚举标签）的项]
```

### 第 3 步 — 设计检查

对照设计规范和用户体验原则检查现有实现，写入 `design-review.md`。

**检查维度：**

**3a. 信息架构**
- 内容分组是否符合用户心智模型（工程师配置工具 vs 数据库实体分组的区别）
- 导航深度是否合理（一般工程工具不超过 3 层）
- 关联实体的配置是否在同一视图内可访问

**3b. 视觉一致性**
- 同类型操作是否使用一致的视觉处理
- 主次操作的视觉权重是否明确
- 数据密集区（表格、曲线、代码块）的信息密度是否适当

**3c. 反馈与状态**
- 加载状态、空状态、错误状态是否都有明确处理
- 保存中 / 生成中的操作进度是否可感知
- 多步骤配置（如 Area → Requirement → Policy → Fan 的关联链）是否有完成度指示

**3d. 工程工具特定检查**
- 魔法值（如 `32768`）是否在 UI 层转换为人类可读文本
- 枚举整数（如 `pid_control_mode: 1`）是否有标签映射展示
- 技术字段名（如 `monitoring_status`、`hardware_pwm`）是否有中文说明或工具提示
- 数组类型字段（如 `temperature_range_low`）的编辑体验是否友好

**真实案例发现 — cooling-web-config 设计问题：**
```
❌ CSR 输出面板 max-height: 160px — 工程师查看生成代码的主要输出区却极为局促
❌ 曲线表格 max-height: 200px — 27 行温度-转速数据在 200px 内滚动，难以全局观察
❌ YAML 编辑器与表单各占 50% 宽度 — 实际使用中用户倾向于只用其中一种，比例应可调
⚠️ 表单网格 grid-template-columns: 1fr 1fr — 在小屏幕下字段过窄，数字输入框宽度不足
⚠️ 「生成CSR」结果直接覆盖输出区 — 没有历史记录，无法对比前后差异
```

**输出格式：**
```markdown
## 问题清单
| # | 维度 | 位置/组件 | 问题描述 | 优先级 | 建议方案 |
|---|------|----------|---------|--------|---------|

## 设计亮点
[记录现有实现中值得保留的设计决策]
```

### 第 4 步 — 设计机会点分析

在审查基础上，主动识别可以提升体验的方向，写入 `design-opportunities.md`。

**分析框架：**

**4a. 效率提升机会**
- 哪些多步操作可以合并或简化
- 哪些重复性配置可以支持批量操作或模板复用
- 哪些字段可以用智能默认值或联动推断减少用户输入

**4b. 认知负担降低机会**
- 哪些技术术语可以用业务语言替换（对操作人员更友好）
- 哪些复杂的配置关系可以用可视化方式呈现（如关系图、依赖树）
- 哪些错误可以提前在输入时预防而非提交后报错

**4c. 工程协作机会**
- 哪些设计改动可以在前端独立实现（不改后端）
- 哪些改动需要后端配合（增加字段、调整数据结构）

**真实案例机会点 — cooling-web-config：**
```
🟢 前端可独立实现：
① 魔法值转换：32768 → 「—」；pid_control_mode 1/2/3/4 → 带图示的选项卡
② CSR 输出面板可调高度（拖拽 resize）
③ 三种调速模式（节能/高性能/低噪）在同一曲线图上叠加对比显示
④ 配置完成度指示：Area → Requirement → Policy → Fan 关联链的步骤进度条
⑤ Excel 导入入口集成到 Web 页面（目前仅在 VSCode 插件中）

🟡 需后端配合：
⑥ 为枚举字段增加 label 映射（避免前端硬编码中文标签）
⑦ 配置版本历史（需后端存储支持）

🔴 产品层决策（需设计师 + PM 对齐）：
⑧ PID 控制模式的文字说明是否足够，还是需要实测数据支撑选择
⑨ 多机型配置共享（fan_board_num > 1）的 UI 如何区分槽位
```

**输出格式：**
```markdown
## 机会点清单
| # | 机会点 | 当前问题 | 建议方向 | 实现难度 | 预期收益 | 是否需要后端配合 |
|---|--------|---------|---------|---------|---------|----------------|

## 优先级建议
[结合难度和收益，给出本轮迭代应优先实现的 Top 3]
```

### 第 5 步 — 生成工程级前端代码

此步骤产出可直接接入开发工程的代码，与 hifi-to-demo 的演示代码有本质区别。

**工程级代码的标准：**
- 组件拆分符合工程实践（单一职责、可复用）
- Props 类型定义完整，字段名与后端数据模型保持完全一致
- 不使用 Mock 数据，通过 Props 或 API 接收真实数据
- 样式方案与目标工程一致（CSS 框架、变量命名）
- 错误处理、加载状态、空状态有完整实现
- 魔法值在数据层统一转换，组件层只处理业务语义

**本项目技术栈：**
JavaScript ES2020+（浏览器原生运行，无需构建工具）+ js-yaml（CDN 加载）+ Canvas（曲线编辑器）

生成代码时严格保持与此技术栈一致，不引入额外依赖。

**枚举值处理规范（来自真实案例）：**
```javascript
// 统一在常量文件中维护，不在组件中硬编码
const COOLING_MODE_LABELS = {
  'EnergySaving':    '节能',
  'LowNoise':        '低噪',
  'HighPerformance': '高性能',
  'Custom':          '自定义',
  'LiquidCooling':   '液冷'
};

const PID_MODE_LABELS = {
  1: '慢升快降（省电优先）',
  2: '快升慢降（响应优先）',
  3: '慢升慢降（稳定优先）',
  4: '快升快降（激进模式）'
};

// 魔法值处理
function formatTemperature(val) {
  return val >= 32768 ? '—' : val + ' ℃';
}
```

### 第 6 步 — 生成接入说明

写入 `integration-guide.md`，帮助开发团队理解如何引入产出代码：

```markdown
# 接入说明

## 前置依赖
[列出需要引入的库，附版本号或 CDN 链接]

## 文件放置
```

## 来自 openUBMC Studio 的实践笔记

**从配置读业务，不只是读字段类型。** `fan_id`、`area_id`、`policy_idx` 这些 ID 背后是整机的调速拓扑关系。读懂它们的关联方式，才能设计出合理的引用选择器 UI。

**魔法值是 UI 层的责任。** `32768` 表示「无效温度」是后端内部约定，UI 绝不能直接展示。这类转换应该集中在 `formatters.js` 里统一处理，而不是散落在各组件中。

**Canvas 组件必须有表格替代方案。** 曲线编辑器用 Canvas 绘制很直观，但 Canvas 内容对辅助技术完全不透明。cooling-web-config 已经提供了曲线表格视图作为备选，这是正确的方向——设计时应确保两种视图完全等价，而非表格只是只读展示。

**工程工具的「信息密度」标准与消费者应用不同。** 专业用户可以处理更高密度的信息，但这不意味着可以用 `font-size: 11px` 或 `max-height: 160px` 的输出区。长时间使用的工具对可读性的要求更高，不是更低。

**可访问性问题根源在后端时，要写清楚。** `pid_control_mode: 1/2/3/4` 这类整数枚举，如果后端不提供 label 映射 API，前端就必须硬编码中文标签——这会导致国际化困难、标签和后端不同步的维护问题。在 accessibility-audit.md 中明确标注，推动后端改进数据结构。

**接入说明写给不了解这段代码历史的人。** 假设读者是刚加入团队的开发，不知道 `32768` 是什么、为什么 `monitoring_status` 是 CSR 同步语法字符串。写清楚这些背景，才是真正有用的接入文档。
