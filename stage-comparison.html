L3 可行性评估
在 openUBMC Studio 场景下，L3「设计主导全链路」模式是否可以实现？
综合判断：L3 在 openUBMC Studio 场景下可实现「前端全链路」，但后端始终需要工程师
部分可行 · 18–24 个月
有利因素
  • CSR 数据结构固定且有文档，设计师可构建高质量 Mock 而无需理解底层硬件
  • 产品以可视化为核心（拓扑图、2.5D 视图），是设计师的天然强项
  • VS Code Extension 的 Webview 架构，设计产出代码可直接在 Studio 运行
  • 用户是工程师，对功能完整度的容忍度高于视觉精美度，降低 Demo 门槛
  • AI 工具在静态页面和可视化组件生成上的能力已足够成熟[1]
核心障碍
  • 后端复杂度（CSR 解析、I²C 总线通信、硬件状态轮询）始终需要专业工程师
  • 生产级代码的性能、安全性、错误处理标准远超 AI Demo 的开箱质量
  • 设计师承担前端架构决策风险，当前团队能力尚未达到此水平
  • 组织评价体系尚未适配「设计师写代码」的工作模式
  • 提示词资产库的系统化建设需要持续投入，是前置条件[5]
可行性结论
  • L3-前端：可行。设计师主导完整前端代码仓 + Mock 数据，开发仅维护后端接口。预计 12–18 个月
  • L3-全栈：短期内不现实。后端和系统层始终是专业工程师领域
  • 当前优先：在 L1 基础上扎实推进 L2，以提示词资产和可复用组件为核心积累
  • 参考：Agentic UX 研究表明协作式半自动化比全自动满意度高 35–50%[1]，L3 并非终极目标
核心判断：L3 不是「设计师替代工程师」，而是「前端归设计，后端归工程」的职责重新划分。对于 openUBMC Studio 这样的可视化密集型工具产品，这种分工天然合理——设计师拥有最强的可视化表达能力，工程师拥有最强的系统设计能力，AI 工具让两者的协作成本趋近于零。

研究洞察 · 对 openUBMC Studio 设计实践的启发
洞察 01 · 代理人格设计空白
设计师需要定义 AI 的工作「人格」，而非只定义 UI 样式
Agentic UX 研究发现，2026年几乎所有产品使用「中性工具人格」，但能定制代理人格的产品用户留存率显著更高[8]。对应到我们的实践：提示词风格、AI 主动程度、面对歧义时的澄清方式——这些「AI 工作方式」的设计，是当前最大的空白也是最大的机会。设计师需要从定义「界面长什么样」扩展到定义「AI 该怎么工作」。
洞察 02 · 效率型 vs 体验型任务
不是所有交付都适合全 AI 生成，设计师要判断任务类型
研究发现「解决速度」不是唯一目标——约 30% 用户在某些场景享受「探索过程」本身[8]。映射到设计交付：批量页面的样式代码（效率型）适合全量 AI 生成；核心可视化组件（体验型）如拓扑图节点设计需要设计师亲自审美把控，保留人工「摩擦」和审美判断。引入「任务类型感知」：效率型全速交 AI，体验型设计师主导迭代。
洞察 03 · 透明度悖论
提示词资产库是解决设计开发信任鸿沟的关键机制
Agentic UX 的核心问题之一：展示太多推理细节导致过载，展示太少又引发不信任。我们的解决方案天然匹配分层透明度模型——提示词资产库对开发开放，开发可以「按需查看」设计师如何描述意图，理解 AI 为何生成这样的代码，而不是被动接收一个黑盒产物。这是建立跨职能信任的最低成本路径。
洞察 04 · 商业模式与交付价值重构
设计师的价值从「交付物数量」转向「解决问题速度与质量」
Agentic UX 将商业模式从「停留时长」重构为「解决速度」，对应到设计团队：设计师的价值不再以「出图数量」衡量，而以「缩短产品落地周期」衡量。L1 阶段的代码 Demo 让开发免去前端实现工作，直接释放了工程资源给后端逻辑——这种价值转移是设计团队向上争取资源和话语权的最有力论据。

  • 从"实现者"转向"接口补全者"和"质量把关者"
  • 视觉还原工作量大幅降低，释放精力给系统架构

  • 前端与后端可以更早解耦、并行推进

  • AI 生成代码的质量与可维护性责任归属不明确

  • 设计师负担增加，需评估是否有充足时间
  • 代码结构规范需要设计 × 开发共同制定

关键发现
01
Skill 的核心价值是结构化交付，不是「能否生成代码」
三个基线 agent 都生成了功能完整的 Demo，但都缺少 design-structure / interaction-spec / tech-selection / prompt-asset 四份文档——这四份才是 L1 协作的核心产出，是设计侧与开发侧共用的工作底稿。

「先看分析再写代码」是非区分性断言
Eval 2 中用户写了明确指令，有无 Skill 都能响应。Skill 的真正价值是在用户没有明确说明时，也主动先做分析确认再出代码，将隐性的好实践变成默认行为。

Skill 增加约 2× 的时间与 Token 成本，代价合理
从 1 个 artifact 扩展为 5 个，平均多用 176s 和 16k token。这些文档直接用于设计 × 开发对齐，对于多轮迭代项目，成本在第二次协作时即可摊销——也是提示词资产沉淀的起点。

## 工作流

### 第 1 步 — 收集输入

如有不明确的地方主动询问：缺失的色彩 Token、边缘状态的交互行为、数据 Schema。
不要猜测会实质影响实现的内容。

同时确认以下两点：
- 是否涉及参考开源代码？若是，记录库名与参考目的，后续生成 `learning/` 文档。
- 是否为团队协作项目？若是，确认分支命名规范偏好（默认提供通用方案）。

### 第 2 步 — 分析设计 · 结构 + 元素描述

**写任何代码之前**，先完成以下分析并写入 `design-structure.md`：

**2a. 布局层级**
- 页面整体分区（顶栏 / 侧边栏 / 主区 / 底部等）
- 各区域尺寸与间距系统

**2b. 组件清单 + 元素详细描述**
对每个独立 UI 组件逐一描述：
- 组件名称与类型（容器 / 展示 / 输入 / 导航等）
- 视觉细节：背景色、边框、圆角、阴影、字体规格
- 内部元素的层叠关系（哪些元素叠在哪些元素上方）
- 与相邻组件的间距关系
- 空状态 / 加载状态下的外观（若设计稿有体现）

**2c. 色彩 Token**
提取所有使用的 hex 值，优先 hex 而非颜色名。

**2d. 数据接口**
每个组件由什么数据驱动？定义 TypeScript 风格的接口。

**2e. 非静态区域标注**
识别设计稿中哪些区域不是静态图：
- 动态数据区（列表、图表、实时数值）
- 加载中 / 骨架屏区域
- 条件渲染区（登录态 / 权限控制 / 空状态）
- 动画 / 过渡区域

若用户明确说「先别急着写代码」「先给我看分析」，**在此停止**，呈现分析后等待确认。

### 第 3 步 — 交互分析 + 可行性评估（合并）

将交互分析与可行性检查合并完成，写入 `interaction-spec.md`。

**3a. 可交互区域与交互动作分析**
对每个可交互元素：
- 交互触发方式（点击 / 悬停 / 拖拽 / 滚动 / 输入 / 手势）
- 触发后的系统响应（页面跳转 / 状态变更 / 弹层 / 展开折叠 / 数据更新）
- 完整状态枚举：default / hover / active / disabled / loading / error / empty / success
- 逆操作路径（用户如何撤销或返回）

**3b. 待交互设计师确认清单**
以下内容从设计稿无法确定，需交互设计师补充或确认后方可实现：
- 在 `interaction-spec.md` 末尾单独列出「⚠️ 待设计师确认」区块
- 典型项目：动画时长与缓动曲线、错误提示文案与出现时机、边缘状态（空数据 / 超长文本 / 权限不足）的视觉处理方式、多步骤流程的回退规则

**3c. 可行性与拓展性评估**
对每个交互动作评估：
- 当前 Demo 阶段能否实现（完整实现 / Mock 实现 / 仅占位）
- 接入真实数据后是否需要重构
- 是否存在影响后续拓展的架构决策（如状态管理方式、事件总线、组件通信模式）

标记以下需要澄清的高风险项：
- 复杂数据可视化（图布局、实时流）
- 需要物理引擎或 Canvas 的动画
- 平台特定 API（摄像头、文件系统、原生手势）

### 第 4 步 — 技术选型

按产品类型选型：

| 产品类型 | 推荐技术栈 |
|---------|-----------|
| 图 / 拓扑可视化 | Vue 3 + ReactFlow 或 Cytoscape.js + ELK/dagre |
| 数据密集型 Dashboard | React + shadcn/ui + Recharts |
| 表单密集型企业工具 | Vue 3 + Element Plus |
| 移动端消费者应用 | Plain HTML+CSS+JS 或 React mobile-first Tailwind |
| SaaS 管理后台 | React + Tailwind CSS |

**利益相关者审查 Demo 始终优先单文件、零安装方案。** 目标是一个双击就能在浏览器打开的文件，而不是需要 `npm install` 的项目。

在 `tech-selection.md` 中记录选型理由、被排除的方案及原因、迁移路径。

### 第 5 步 — 生成代码

分层构建：
1. 静态 HTML 结构（正确的语义元素）
2. 应用样式（颜色 / 字体 / 间距与设计稿对齐）
3. 添加交互（状态变更、过渡动画、事件处理器）
4. 接入 Mock 数据（字段命名真实，表格至少 5–8 条记录）

### 第 6 步 — 初步易用性审查

代码生成后，在提交给用户之前先自查，将结果写入 `quality-review.md`。

**审查维度：**

| 维度 | 检查项 |
|------|--------|
| 可读性 | 文字对比度是否足够（深色背景配浅色文字 / 反之）；字号是否过小（正文 < 12px 即警告） |
| 层级清晰度 | 主要操作按钮与次要按钮是否有明显视觉权重差异；标题与正文是否可区分 |
| 交互反馈 | 可点击元素是否有 hover / active 状态；按钮 disabled 时是否视觉上明显不可点 |
| 布局稳定性 | 动态内容填入后是否会引起意外布局抖动；长文本是否有截断或换行处理 |
| 空状态 | 列表 / 图表为空时是否有提示，而非空白区域 |
| 错误状态 | 表单错误提示是否清晰、位置是否靠近出错字段 |

对每个问题标注：
- ✅ 已在 Demo 中修复
- ⚠️ 存在但不影响核心流程（说明原因）
- ❌ 需设计师重新定义（说明为什么无法在 Demo 层修复）

## 来自 openUBMC Studio 的实践笔记

**Hex 优于颜色名。** 始终从设计稿提取精确 hex 值。`#141420` 无歧义；「深海军蓝」则不然。

**图布局必须用布局引擎。** 永远不要硬编码图节点的 x/y 坐标——哪怕只有 5 个节点也会变成维护噩梦。使用 ELK、dagre 或 ReactFlow 自动布局。若无法通过 npm 获取布局引擎，改用按层编程定位（如 `computeLayout()` 按层分组），并记录升级路径。

**Mock 数据 = API 契约。** Mock 数据的字段形状应与真实后端返回保持一致。字段命名如 `lastLoginAt`（而非 `last_login`）应匹配约定的 API 规范，这样 Demo 可直接用于 API 契约协商。

**暂停并确认。** 当用户要求先看分析再写代码时，这是工作流信号——他们想在投入前验证理解。要真正停下来呈现结构化分析，而不是一边「确认」一边继续写代码。

**交互设计师确认是关口，不是可选项。** `interaction-spec.md` 中的「待确认」清单不是礼貌性提问，是真正的实现前置条件。没有确认就写的交互细节大概率要返工。

**文档是协作界面。** README / CHANGELOG / BRANCH 不是形式主义——它们是设计侧产出与开发侧接收之间的协议文件。写清楚比写漂亮更重要。

# 后端代码 → 设计反向分析 · 工程级代码生成

本 skill 对应 openUBMC Studio L1 协作中「后端先行」场景：
后端交付代码或配置后，设计师不是从零出发，而是以代码为锚点，
反向建立设计规范、发现问题、生成可直接接入工程的前端代码。

## 工作流

### 第 1 步 — 读懂后端交付物

接收并解析后端交付的所有内容，写入 `code-reading.md`：

**1a. 识别交付物类型**
- 配置文件（YAML / JSON / TOML）：提取字段结构、枚举值、约束条件
- 数据模型文档（Markdown / OpenAPI）：提取实体关系、字段类型、必填项
- 源代码（Vue / React / 原生 JS）：提取组件结构、状态管理、事件处理
- API 文档：提取端点、请求/响应结构、错误码

**1b. 建立数据字典**
对每个关键字段记录：
- 字段名（原始命名）
- 数据类型与取值范围
- 业务含义（中文说明）
- 与界面的对应关系（这个字段驱动哪个 UI 元素）

**1c. 梳理业务逻辑**
- 核心流程：用户/系统的主要操作路径是什么
- 状态机：数据有哪些状态，状态之间如何转换
- 约束规则：哪些字段联动、哪些值触发特殊行为

若有不明确的字段含义或业务逻辑，**主动向用户确认**，不要猜测。

### 第 2 步 — 可访问性审查

基于对代码/配置的理解，写入 `accessibility-audit.md`。

**审查维度（对照 WCAG 2.1 AA 标准）：**

| 维度 | 检查项 |
|------|--------|
| 可感知性 | 色彩对比度是否 ≥ 4.5:1（文字）/ 3:1（大号文字和界面组件）；非文字信息是否有替代文本 |
| 可操作性 | 所有功能是否可通过键盘完成；焦点顺序是否符合视觉顺序；操作时间限制是否合理 |
| 可理解性 | 错误提示是否描述了问题且给出修复建议；表单标签是否与输入框明确关联 |
| 鲁棒性 | 是否使用了语义化 HTML；ARIA 属性使用是否正确 |
| 工程特定项 | 配置类界面：必填项是否有明确标注；枚举值是否有人类可读的标签（不只是机器值） |

**输出格式：**
```
## 问题清单
| # | 维度 | 问题描述 | 严重程度 | 受影响用户 | 修复建议 |
|---|------|---------|---------|-----------|---------|
| 1 | 可感知性 | ... | 严重/重要/建议 | ... | ... |

## 已符合项
[列出已做得好的可访问性实践，避免后续迭代破坏]
```

### 第 3 步 — 设计检查

对照设计规范和用户体验原则检查现有实现，写入 `design-review.md`。

**检查维度：**

**3a. 信息架构**
- 内容分组是否符合用户心智模型（而非技术/数据库分组）
- 层级深度是否合理（一般不超过 3 层）
- 导航路径是否可预测

**3b. 视觉一致性**
- 同类元素是否使用一致的视觉处理（颜色、间距、字体权重）
- 交互元素（按钮、输入框、下拉）是否有统一的样式规范
- 数据密集型界面：表格、列表、卡片的信息密度是否适当

**3c. 反馈与状态**
- 加载状态、空状态、错误状态是否都有明确的视觉处理
- 操作结果（成功/失败）是否及时反馈
- 配置类场景特别检查：保存中、保存成功、验证失败的状态是否清晰

**3d. 工程代码特定检查**
- 硬编码字符串是否影响国际化
- 配置字段的默认值展示是否对用户友好
- 枚举值（如 `status: 0/1/2`）是否有可读的映射展示

**输出格式：**
```
## 问题清单
| # | 维度 | 位置/组件 | 问题描述 | 优先级 | 建议方案 |
|---|------|----------|---------|--------|---------|

## 设计亮点
[记录现有实现中值得保留的设计决策]
```

### 第 4 步 — 设计机会点分析

在审查基础上，主动识别可以提升体验的方向，写入 `design-opportunities.md`。

**分析框架：**

**4a. 效率提升机会**
- 哪些多步操作可以合并或简化
- 哪些重复性配置可以支持批量操作或模板
- 哪些字段填写可以用智能默认值或联动推断减少用户输入

**4b. 认知负担降低机会**
- 哪些技术术语可以用业务语言替换（对操作人员更友好）
- 哪些复杂的配置关系可以用可视化方式呈现（如关系图、依赖树）
- 哪些错误可以提前在输入时预防而非提交后报错

**4c. 工程协作机会**
- 哪些设计改动可以在不修改后端的前提下在前端实现
- 哪些改动需要后端配合（增加字段、调整数据结构）
- 哪些可以通过配置驱动的方式支持不同团队的定制需求

**输出格式：**
```
## 机会点清单
| # | 机会点 | 当前问题 | 建议方向 | 实现难度 | 预期收益 | 是否需要后端配合 |
|---|--------|---------|---------|---------|---------|----------------|

## 优先级建议
[结合难度和收益，给出本轮迭代应优先实现的 Top 3]
```

### 第 5 步 — 生成工程级前端代码

与 hifi-to-demo 的 Demo 代码不同，此步骤产出的是**可直接接入开发工程**的代码。

**工程级代码的标准：**
- 组件拆分符合工程实践（单一职责、可复用、可测试）
- Props 类型定义完整，与后端数据模型字段名保持一致
- 不使用硬编码 Mock 数据，通过 Props 或 API 调用接收真实数据
- 样式方案与目标工程一致（使用相同的 CSS 框架或 CSS-in-JS 方案）
- 错误处理、加载状态、空状态有完整实现
- 关键交互有注释说明设计意图

**代码生成策略：**

| 目标工程技术栈 | 生成方案 |
|-------------|---------|
| Vue 3 + Element Plus | Composition API + `<script setup>` + Element Plus 组件 |
| Vue 3 + 自定义组件库 | Composition API + 原子样式（若工程用 Tailwind/UnoCSS） |
| React + Ant Design | FC + Hooks + Ant Design 组件 |
| 纯 HTML/JS（工具类页面） | 语义化 HTML + CSS 自定义属性 + 原生 JS |

**输出目录结构：**
```
frontend/
├── components/
│   ├── [ComponentName].vue (或 .tsx)
│   └── ...
├── types/
│   └── index.ts          # 与后端数据模型对应的 TS 接口
├── utils/
│   └── [helpers].ts      # 数据转换、格式化等辅助函数
└── README.md             # 组件使用说明
```

**特别要求：**
- `types/index.ts` 中的接口字段名必须与后端数据模型保持一致
- 枚举值使用 `const enum` 或常量对象，附带中文说明
- 日期、数值格式化函数单独抽离，方便后续国际化

## 来自 openUBMC Studio 的实践笔记

**从代码读业务，不只是读技术。** 配置文件中每个字段背后都有业务决策。读懂字段的含义比读懂它的类型更重要——`fan_speed_mode: 0/1/2` 对用户来说是「自动/手动/节能」，而不是一个整数。

**工程代码与 Demo 代码的根本区别。** Demo 代码证明「可以做」，工程代码是「直接做」。工程代码要承担错误处理、类型安全、可维护性的责任，不能依赖 Mock 数据蒙混过关。

**可访问性是与后端的协作问题。** 很多可访问性问题（如枚举值缺少人类可读标签、错误码缺少友好说明）根源在后端数据结构，不能只靠前端修补。在 `accessibility-audit.md` 中明确标注哪些问题需要后端配合。

**设计机会点要区分「前端能做」和「需要后端」。** 前端能做的改动不需要等待排期，优先推进。需要后端的改动要提前与后端沟通，写清楚需要增加什么字段或接口。

**接入说明写给不了解这段代码历史的人。** 写的时候假设读者是刚加入团队、不知道这段代码背景的开发，这样的接入说明才真正有用。
