L3 可行性评估
在 openUBMC Studio 场景下，L3「设计主导全链路」模式是否可以实现？
综合判断：L3 在 openUBMC Studio 场景下可实现「前端全链路」，但后端始终需要工程师
部分可行 · 18–24 个月
有利因素
  • CSR 数据结构固定且有文档，设计师可构建高质量 Mock 而无需理解底层硬件
  • 产品以可视化为核心（拓扑图、2.5D 视图），是设计师的天然强项
  • VS Code Extension 的 Webview 架构，设计产出代码可直接在 Studio 运行
  • 用户是工程师，对功能完整度的容忍度高于视觉精美度，降低 Demo 门槛
  • AI 工具在静态页面和可视化组件生成上的能力已足够成熟[1]
核心障碍
  • 后端复杂度（CSR 解析、I²C 总线通信、硬件状态轮询）始终需要专业工程师
  • 生产级代码的性能、安全性、错误处理标准远超 AI Demo 的开箱质量
  • 设计师承担前端架构决策风险，当前团队能力尚未达到此水平
  • 组织评价体系尚未适配「设计师写代码」的工作模式
  • 提示词资产库的系统化建设需要持续投入，是前置条件[5]
可行性结论
  • L3-前端：可行。设计师主导完整前端代码仓 + Mock 数据，开发仅维护后端接口。预计 12–18 个月
  • L3-全栈：短期内不现实。后端和系统层始终是专业工程师领域
  • 当前优先：在 L1 基础上扎实推进 L2，以提示词资产和可复用组件为核心积累
  • 参考：Agentic UX 研究表明协作式半自动化比全自动满意度高 35–50%[1]，L3 并非终极目标
核心判断：L3 不是「设计师替代工程师」，而是「前端归设计，后端归工程」的职责重新划分。对于 openUBMC Studio 这样的可视化密集型工具产品，这种分工天然合理——设计师拥有最强的可视化表达能力，工程师拥有最强的系统设计能力，AI 工具让两者的协作成本趋近于零。

研究洞察 · 对 openUBMC Studio 设计实践的启发
洞察 01 · 代理人格设计空白
设计师需要定义 AI 的工作「人格」，而非只定义 UI 样式
Agentic UX 研究发现，2026年几乎所有产品使用「中性工具人格」，但能定制代理人格的产品用户留存率显著更高[8]。对应到我们的实践：提示词风格、AI 主动程度、面对歧义时的澄清方式——这些「AI 工作方式」的设计，是当前最大的空白也是最大的机会。设计师需要从定义「界面长什么样」扩展到定义「AI 该怎么工作」。
洞察 02 · 效率型 vs 体验型任务
不是所有交付都适合全 AI 生成，设计师要判断任务类型
研究发现「解决速度」不是唯一目标——约 30% 用户在某些场景享受「探索过程」本身[8]。映射到设计交付：批量页面的样式代码（效率型）适合全量 AI 生成；核心可视化组件（体验型）如拓扑图节点设计需要设计师亲自审美把控，保留人工「摩擦」和审美判断。引入「任务类型感知」：效率型全速交 AI，体验型设计师主导迭代。
洞察 03 · 透明度悖论
提示词资产库是解决设计开发信任鸿沟的关键机制
Agentic UX 的核心问题之一：展示太多推理细节导致过载，展示太少又引发不信任。我们的解决方案天然匹配分层透明度模型——提示词资产库对开发开放，开发可以「按需查看」设计师如何描述意图，理解 AI 为何生成这样的代码，而不是被动接收一个黑盒产物。这是建立跨职能信任的最低成本路径。
洞察 04 · 商业模式与交付价值重构
设计师的价值从「交付物数量」转向「解决问题速度与质量」
Agentic UX 将商业模式从「停留时长」重构为「解决速度」，对应到设计团队：设计师的价值不再以「出图数量」衡量，而以「缩短产品落地周期」衡量。L1 阶段的代码 Demo 让开发免去前端实现工作，直接释放了工程资源给后端逻辑——这种价值转移是设计团队向上争取资源和话语权的最有力论据。

✦ 设计师角色扩展

  • 设计稿不再是唯一交付物，代码 Demo 成为新交付标准
  • 提示词工程成为核心技能（等同于设计规范撰写能力）
  • 技术选型发言权从"建议"变为"决定"
  • 掌握产品验证权：不再依赖开发才能做用研原型
  • 需要建立数据结构和接口设计的基础认知（L3）

⟳ 开发角色转变

  • 从"实现者"转向"接口补全者"和"质量把关者"
  • 视觉还原工作量大幅降低，释放精力给系统架构
  • 需要具备"接手 AI 生成代码"的能力（评审/重构）
  • 前端与后端可以更早解耦、并行推进

⚠ 新增摩擦点 / 风险

  • AI 生成代码的质量与可维护性责任归属不明确
  • 设计师代码与开发代码的"翻译成本"需要规范消化
  • 提示词资产如不沉淀，每次迭代成本会反弹
  • 设计师负担增加，需评估是否有充足时间
  • 代码结构规范需要设计 × 开发共同制定

关键发现
01
Skill 的核心价值是结构化交付，不是「能否生成代码」
三个基线 agent 都生成了功能完整的 Demo，但都缺少 design-structure / interaction-spec / tech-selection / prompt-asset 四份文档——这四份才是 L1 协作的核心产出，是设计侧与开发侧共用的工作底稿。
02
「先看分析再写代码」是非区分性断言
Eval 2 中用户写了明确指令，有无 Skill 都能响应。Skill 的真正价值是在用户没有明确说明时，也主动先做分析确认再出代码，将隐性的好实践变成默认行为。
03
Skill 增加约 2× 的时间与 Token 成本，代价合理
从 1 个 artifact 扩展为 5 个，平均多用 176s 和 16k token。这些文档直接用于设计 × 开发对齐，对于多轮迭代项目，成本在第二次协作时即可摊销——也是提示词资产沉淀的起点。

# 高保真设计稿 → Demo → 工程代码 两阶段工作流

本 skill 提炼自 openUBMC Studio L1 协作实践。
核心理念：**Demo 是验证工具，不是最终交付物。**
设计意图通过 Demo 快速对齐之后，工程代码才是真正能进仓库的资产。

---

## 需求分析

### 用户

| 角色 | 使用场景 | 关注点 |
|------|---------|--------|
| 体验设计师 | 将设计稿转化为可演示的原型或工程代码 | 设计意图能被准确还原；交互细节不被遗漏 |
| 前端开发工程师 | 接收设计侧的工程代码，直接集成进项目 | 代码规范一致；类型完整；能直接跑起来 |
| 产品经理 / 利益相关者 | 评审 Demo 原型，确认设计方向 | 能在浏览器里直接看，不需要安装任何东西 |
| 交互设计师 | 审阅交互细化文档，补充确认清单 | 状态枚举完整；边缘情况有明确提案 |

### 场景

1. **快速原型验证**：设计稿刚完成，需要在利益相关者评审前做可点击原型——Demo 阶段，单文件，零安装。
2. **设计交付开发**：设计方向已对齐，开发需要能直接集成的组件代码——工程阶段，类型完整、状态覆盖。
3. **多轮迭代**：Demo 评审后有修改意见，在 Phase 1 内快速迭代，方向稳定后再生成工程代码，避免在工程代码上来回改。
4. **设计资产沉淀**：不只是代码，还有结构文档、交互说明、质量报告，形成可复用的协作资产。

### 解决的问题

- **设计-开发鸿沟**：设计稿是静态图，开发理解偏差导致实现走样。通过 `design-structure.md` 将设计意图结构化，消除歧义。
- **交互细节丢失**：高保真图无法表达状态枚举和边缘情况。`interaction-spec.md` 的「待确认清单」把隐式假设显式化，在实现前对齐。
- **Demo 代码进工程**：Demo 代码将就、无类型、Mock 数据硬编码，直接进工程会留下技术债。两阶段明确分工，各自达到各自的质量标准。
- **方向未定就写工程代码**：在设计还在迭代时就写工程级代码，一旦方向变了返工成本极高。Phase 1 → 确认节点 → Phase 2 的流程从结构上避免了这个问题。

### 设计点

- **两阶段分离**：Demo 优化「能不能看」，工程代码优化「能不能用」，目标不同、标准不同，不能合并成一次交付。
- **Mock 数据即 API 契约**：Demo 的字段命名从 Phase 1 就对齐后端约定，Phase 2 的 API 层是替换而非重写。
- **确认节点是真正的关口**：Phase 1 完成后的确认不是礼节，是最便宜的变更窗口——Demo 层改比工程层改快得多。
- **文档是协作界面**：README / CHANGELOG / BRANCH / INTEGRATION 不是形式主义，是设计侧产出与开发侧接收之间的契约文件。

## Phase 1 工作流 — Demo 快速验证

### 第 1 步 — 收集输入

主动询问不明确的地方：缺失的色彩 Token、边缘状态的交互行为、数据 Schema。
不要猜测会实质影响实现的内容。

同时确认：
- 是否涉及参考开源代码？若是，记录库名，后续生成 `learning/` 文档。
- 是否为团队协作项目？若是，确认分支命名规范偏好。
- **目标开发栈是什么？**（React / Vue 3 / 原生 HTML，框架版本，CSS 方案）—— 工程代码需要对齐，Phase 1 阶段提前了解。

### 第 2 步 — 分析设计 · 结构 + 元素描述

**写任何代码之前**，先完成分析并写入 `design-structure.md`：

**2a. 布局层级**
- 页面整体分区（顶栏 / 侧边栏 / 主区 / 底部等）
- 各区域尺寸与间距系统

**2b. 组件清单 + 元素详细描述**
对每个独立 UI 组件逐一描述：
- 组件名称与类型（容器 / 展示 / 输入 / 导航等）
- 视觉细节：背景色、边框、圆角、阴影、字体规格（均用 hex 值）
- 内部元素层叠关系
- 与相邻组件的间距关系
- 空状态 / 加载状态下的外观

**2c. 色彩 Token** — 提取所有 hex 值

**2d. 数据接口** — TypeScript 风格接口定义，字段命名与预期后端 API 保持一致

**2e. 非静态区域标注**
- 动态数据区 / 骨架屏区域 / 条件渲染区 / 动画区

若用户要求「先看分析」，在此暂停并等待确认。

### 第 3 步 — 交互分析 + 可行性评估

写入 `interaction-spec.md`：

**3a. 可交互区域**
对每个可交互元素：触发方式 → 系统响应 → 状态枚举 → 逆操作路径

**3b. 待设计师确认清单**
在文件末尾单独列「⚠️ 待设计师确认」区块：
动画时长与缓动曲线 / 错误提示文案与时机 / 边缘状态视觉处理 / 多步骤流程回退规则

**3c. 可行性与拓展性评估**
- Demo 阶段能否实现（完整 / Mock / 占位）
- 接入真实数据后是否需要重构
- 影响后续拓展的架构决策

### 第 4 步 — 技术选型（两套）

在 `tech-selection.md` 中分别记录：

**Demo 选型原则**：单文件、零安装、双击即开。
CDN 引入框架，Mock 数据内联，不依赖构建工具。

**工程选型原则**：对齐用户确认的目标开发栈。

| 产品类型 | 推荐工程技术栈 |
|---------|--------------|
| 图 / 拓扑可视化 | Vue 3 + ReactFlow 或 Cytoscape.js + ELK/dagre |
| 数据密集型 Dashboard | React + shadcn/ui + Recharts + TanStack Query |
| 表单密集型企业工具 | Vue 3 + Element Plus + Pinia |
| 移动端消费者应用 | React + Tailwind CSS + React Query |
| SaaS 管理后台 | React + Tailwind CSS + Zustand |
| BMC / 嵌入式管理界面 | 原生 HTML + CSS + JS（无框架，保持轻量） |

记录：Demo 到工程的迁移路径、被排除的方案及原因。

### 第 5 步 — 生成 Demo 代码

分层构建 `demo/index.html`：
1. 静态 HTML 结构（正确的语义元素）
2. 样式（颜色 / 字体 / 间距与设计稿对齐）
3. 交互（状态变更、过渡动画、事件处理器）
4. Mock 数据（字段命名与真实 API 约定一致，至少 5–8 条记录）

### 第 6 步 — 易用性审查

代码生成后先自查，写入 `quality-review.md`：

| 维度 | 检查项 |
|------|--------|
| 可读性 | 文字对比度；字号（正文 < 12px 即警告） |
| 层级清晰度 | 主次按钮视觉权重差异；标题与正文可区分 |
| 交互反馈 | hover / active / disabled 状态明确 |
| 布局稳定性 | 动态内容填入后无布局抖动；长文本有截断处理 |
| 空状态 | 列表 / 图表为空时有提示，非空白区域 |
| 错误状态 | 表单错误提示清晰、位置靠近出错字段 |

标注每个问题：✅ 已修复 / ⚠️ 已知但不阻碍核心流程 / ❌ 需设计师重新定义

## Phase 2 工作流 — 工程级代码

> 前提：Phase 1 Demo 已通过评审，用户已确认设计方向和目标开发栈。

进入 Phase 2 前，主动了解：
- 目标仓库的目录结构约定（`components/` 在哪？API 层在哪？）
- 状态管理方案（Redux / Pinia / Zustand / 无）
- CSS 方案（CSS Modules / Tailwind / styled-components / BEM）
- 测试框架（Vitest / Jest / 无）
- TypeScript 严格度（strict 模式 / 宽松 / 纯 JS）

### 第 7 步 — 组件拆分与工程结构规划

根据 Demo 和目标仓库约定，规划组件树：

```
src/
├── components/
│   └── [FeatureName]/
│       ├── index.tsx           # 组件入口
│       ├── types.ts            # Props / Events / 共享类型
│       ├── [SubComponent].tsx  # 子组件（如有）
│       └── [FeatureName].test.ts
├── services/
│   └── [feature]Api.ts         # API 集成层
└── (hooks / store / utils — 视项目结构而定)
```

在开始写代码之前，把组件树和文件结构告诉用户确认——这一步返工成本最高。

### 第 8 步 — 生成工程组件

**组件质量标准：**

- **TypeScript 严格**：Props 类型完整，无 `any`，Emits / Events 有类型声明
- **无硬编码**：颜色、文字、枚举值提取为常量或 Token，不内联在 JSX/模板里
- **状态全覆盖**：loading / error / empty / disabled 状态均有对应 UI，不遗漏
- **无副作用泄漏**：useEffect 有清理函数；Vue 组件在 onUnmounted 移除监听器
- **可访问性（a11y）**：ARIA 标签、键盘导航、焦点管理符合 WCAG 2.1 AA 基线
- **无直接 DOM 操作**：不用 `document.querySelector`，用框架的 ref / $el 机制
- **CSS 不污染全局**：CSS Modules / scoped / styled-components，类名有命名空间
- **错误边界**：关键数据加载失败时有 fallback UI，不白屏

### 第 9 步 — API 集成层

将 Demo 的 Mock 数据替换为真实 API 调用骨架，写入 `src/services/[feature]Api.ts`：

```typescript
// 保留 Mock 作为开发降级，用环境变量或 flag 切换
const USE_MOCK = import.meta.env.VITE_USE_MOCK === 'true'

export async function fetchFeatureData(params: QueryParams): Promise<FeatureData[]> {
  if (USE_MOCK) return mockFeatureData
  const res = await http.get('/api/feature', { params })
  return res.data
}
```

字段命名严格对齐 Phase 1 中 `design-structure.md` 定义的数据接口——这是 Mock 数据即 API 契约的落地。

### 第 10 步 — 单元测试桩

为每个组件生成测试桩，覆盖核心状态分支：

```typescript
describe('[ComponentName]', () => {
  it('renders loading state', () => { /* ... */ })
  it('renders error state', () => { /* ... */ })
  it('renders empty state', () => { /* ... */ })
  it('renders data correctly', () => { /* ... */ })
  it('handles user interaction: [主要交互]', () => { /* ... */ })
})
```

测试桩标注 `// TODO: implement` 而非空 it 块，让开发者知道要补什么。

### 第 11 步 — 接入指南与文档更新

**INTEGRATION.md** — 告诉开发者怎么把代码接进去：

```markdown

[列出依赖的包，带版本号]

4. 替换 Mock 数据：将 VITE_USE_MOCK=true 改为 false，配置真实 API baseURL

## 注意事项
[魔法值、枚举映射、特殊处理逻辑的说明]

## 已知限制
[从 quality-review.md 和 interaction-spec.md 摘取的待处理项]

## 来自 openUBMC Studio 的实践笔记

**Demo 是便宜的实验，工程代码是昂贵的承诺。**
不要在设计方向未确认时就写工程代码——Demo 阶段暴露的问题修起来是分钟级，工程代码里修是小时级。

**Hex 优于颜色名。** 始终从设计稿提取精确 hex 值。`#141420` 无歧义；「深海军蓝」则不然。

**图布局必须用布局引擎。** 永远不要硬编码图节点的 x/y 坐标。使用 ELK / dagre / ReactFlow 自动布局，或按层编程定位并记录升级路径。

**Mock 数据 = API 契约。** Demo 里的 Mock 字段形状应与真实后端返回一致。`lastLoginAt`（而非 `last_login`）命名从 Phase 1 开始就要对齐，这样 Phase 2 的 API 层是替换而非重写。

**组件拆分在写代码前确认。** 组件树比代码实现更容易改。把结构告诉用户对齐后再动手，比写完再重构省力得多。

**接入指南是工程交付的最后一公里。** `INTEGRATION.md` 写得不清楚，开发接收时会在环境配置和依赖版本上卡很久。具体写：包名 + 版本、目录位置、环境变量名称。

**暂停并确认。** Phase 1 到 Phase 2 的节点是真正的关口，不是形式。没有对齐方向就写工程代码，返工成本极高。

# 后端代码 → 设计反向分析 · 工程级代码生成

本 skill 对应 openUBMC Studio L1 协作中「后端先行」场景：
后端交付代码或配置后，设计师不是从零出发，而是以代码为锚点，
反向建立设计规范、发现问题、生成可直接接入工程的前端代码。

---

## 实际走过的完整协同流程

本 skill 来自 openUBMC 蓝区一次完整的「后端先行」协同实践，以下是实际经历的五步：

| 阶段 | 输入 | 工具/方法 | 产出 |
|------|------|----------|------|
| **Step 1** 接收后端交付物 | 代码仓：`cooling-web-config`、`data-model.md`、`sample-cooling.yaml`、`cooling-excel-parser` | 阅读 + 梳理 | 理解产品是什么、做什么 |
| **Step 2** 读代码 → 产品文档 | 以上所有后端交付物 | `/code-to-product-doc`（独立 skill） | 功能定义、用户场景、数据结构、核心流程、设计机会点、注意事项 |
| **Step 3** 产品文档 → 设计审查 | Step 2 的产品文档作为上下文 | `/design-critique` + `/accessibility-review` | 分级问题清单（🔴 Critical / 🟡 Moderate / 🟢 Minor）+ WCAG 2.1 AA 对照报告 |
| **Step 4** 审查结论 → 工程实现 | 审查报告 + 后端数据结构 | Edit/Write 组件代码 | 吸顶导航、可收起面板、步骤引导、静态 HTML 工程级代码，部署到 GitHub Pages |
| **Step 5** 验证与迭代 | 截图反馈 | 排查 → 修复 → 再验证 | 可用的线上 Demo |

> Step 2（代码 → 产品文档）是最容易被省略、但最不该省略的一步。它是后续所有设计决策的基础，也是设计师和工程师建立共同语言的节点。该步骤已提炼为独立 skill：`/code-to-product-doc`。

## 需求分析

### 用户

| 角色 | 使用场景 | 关注点 |
|------|---------|--------|
| 体验设计师 | 收到后端代码后介入，做设计分析和质量检查 | 能读懂代码里的业务含义；发现问题有据可依 |
| 前端开发工程师 | 接收设计侧输出的工程代码，直接集成 | 代码和后端数据结构严格对齐；不引入新的技术债 |
| 后端工程师 | 查看设计侧的可访问性和设计问题反馈 | 问题描述精准，能明确哪些需要后端配合改 |
| 项目负责人 | 评估设计介入后的改进方向和优先级 | 机会点有明确的难度和收益评估 |

### 场景

1. **后端先行交付**：后端代码或配置先于设计稿完成，设计师以代码为一手资料，反向建立对产品的理解。
2. **工程工具设计**：面向专业用户的配置工具、管理后台，没有消费端那样的视觉设计规范，但有更严格的可访问性和效率要求。
3. **设计质量审查**：开发已有现成实现，设计师作为质量守门员介入，找问题、提改进方向，而非全部推翻重做。
4. **设计-开发共同语言建立**：设计师和后端工程师通过代码中的字段命名、数据结构建立共同词汇表，减少沟通摩擦。

### 解决的问题

- **设计师看不懂代码**：代码里有大量对设计工作有价值的信息（枚举值、魔法值、实体关系），但设计师缺少读取和整理这些信息的方法。第 1 步「读懂后端交付物」提供了系统的拆解方式。
- **可访问性问题在后期才暴露**：字号 11px、颜色对比不足、Canvas 无键盘支持——这些问题如果在代码层面早发现，比产品上线后再改成本低得多。
- **设计建议缺乏工程依据**：「这里不好看」和「这里 font-size: 11px 低于 WCAG 2.1 AA 建议的 14px 等效最小值」，后者更容易被工程团队接受和排期。
- **设计改动不知道影响范围**：哪些改动设计师自己能做（前端独立实现），哪些需要后端配合，哪些需要产品层决策——机会点分析把这三类明确分开。

### 设计点

- **从代码提取业务语言**：`pid_control_mode: 1/2/3/4` 本身毫无意义，设计师的工作是将它翻译成用户能理解的「慢升快降（省电优先）」——这个翻译过程就是设计价值。
- **魔法值是 UI 层的责任**：`32768` 是后端内部约定，UI 绝不能直接展示。设计分析的目标之一就是发现这类「后端约定泄漏到用户界面」的问题。
- **可访问性先于视觉设计**：对工程工具类界面，可访问性不是锦上添花，是基础要求——专业用户长时间使用，字号和对比度的问题会在每天 8 小时里持续伤害效率。
- **分层推进，不全推翻**：区分「前端独立实现」「需后端配合」「需产品决策」三类机会点，让改进建议可落地，而不是理想化的全部重做。

## 工作流

### 第 1 步 — 读懂后端交付物

接收并解析后端交付的所有内容，写入 `code-reading.md`：

**1a. 识别交付物类型**
- 配置文件（YAML / JSON / TOML）：提取字段结构、枚举值、约束条件、魔法默认值
- 数据模型文档（Markdown / OpenAPI）：提取实体关系、字段类型、必填项、校验规则
- 源代码（HTML + CSS + JS）：提取组件结构、状态管理、事件处理，注意字号/颜色硬编码
- VSCode 插件或 CLI 工具：理解用户的实际工作上下文（在哪个工具里操作）

**1b. 建立数据字典**
对每个关键字段记录：
- 字段名（原始命名）与数据类型
- 业务含义（用操作人员能理解的语言，而非工程术语）
- 取值范围与枚举映射（如 `pid_control_mode: 1/2/3/4` → 「慢升快降/快升慢降/慢升慢降/快升快降」）
- 魔法默认值标注（如 `32768` 表示「无效/不可用」，对用户不友好，需在 UI 层转换）
- 与界面元素的对应关系

**真实案例参考 — cooling-web-config 字段梳理：**
```
smart_cooling_mode: "EnergySaving" | "LowNoise" | "HighPerformance" | "Custom" | "LiquidCooling"
  → UI 应展示：节能 / 低噪 / 高性能 / 自定义 / 液冷
  → 切换模式会联动影响：target_temperature_celsius 的有效范围

pid_control_mode: 1 | 2 | 3 | 4
  → UI 应展示带图解的选项：慢升快降（省电优先）/ 快升慢降（响应优先）/ ...
  → 整数值对用户完全无意义，必须在 UI 层做映射

sys_hdds_max_temperature: 32768  ← 魔法值
  → 32768 是「无效/获取不到」的内部标记，UI 应显示「—」或「不可用」
  → 绝对不能把 32768 直接展示给用户
```

**1c. 梳理实体关系**
- 哪些实体之间有 1:N 关系（如 `CoolingArea` 引用 `CoolingRequirement` + `CoolingPolicy` + `CoolingFan`）
- 哪些 ID 是「本地标识」vs「全局唯一 ID」（影响引用方式）
- 哪些字段使用 CSR 动态语法（如 `<=/Scanner_Lm75_Inlet.Value`），UI 层需要特殊处理

若有不明确的字段含义或业务逻辑，**主动向用户确认**，不要猜测。

### 第 2 步 — 可访问性审查

基于对代码/配置的理解，写入 `accessibility-audit.md`。
重点关注工程工具类界面（不同于消费者 app，此类工具的用户是专业人员，但更长时间使用）。

**审查维度（对照 WCAG 2.1 AA）：**

| 维度 | 检查项 |
|------|--------|
| 可感知性 | 色彩对比度是否 ≥ 4.5:1（正文）/ 3:1（大字体和 UI 组件）；错误状态是否仅依赖颜色区分 |
| 可操作性 | 所有功能是否可通过键盘完成；Canvas 类组件（如曲线编辑器）是否提供键盘 / 表格替代操作 |
| 可理解性 | 错误提示是否描述了「什么错了」和「如何修复」；技术字段名是否有人类可读标签 |
| 鲁棒性 | 是否使用语义化 HTML；ARIA 属性是否正确；表单 label 是否与 input 关联 |

**真实案例发现 — cooling-web-config 可访问性问题：**
```
❌ font-size: 12px 用于表单 label（低于 WCAG 建议的 14px 等效最小值）
❌ font-size: 11px 用于字段错误消息（几乎不可读）
❌ .field-error 仅用红色边框 + 背景区分，无图标，对色觉障碍用户不友好
❌ Canvas 曲线编辑器（curve-editor.js）无键盘操作支持，仅靠鼠标拖拽
❌ 错误列表 #error-list 没有 role="alert" 或 aria-live，屏幕阅读器无法感知
⚠️ cursor: crosshair 用于曲线画布 — 暗示「绘制」而非「数据编辑」，语义不准确
```

**输出格式：**
```markdown
## 问题清单
| # | 维度 | 位置/组件 | 问题描述 | 严重程度 | 受影响用户 | 修复建议 | 需后端配合 |
|---|------|----------|---------|---------|-----------|---------|----------|

## 已符合项
[记录已做好的可访问性实践，避免后续迭代破坏]

## 待后端配合项
[需要后端增加字段（如人类可读枚举标签）的项]
```

### 第 3 步 — 设计检查

对照设计规范和用户体验原则检查现有实现，写入 `design-review.md`。

**检查维度：**

**3a. 信息架构**
- 内容分组是否符合用户心智模型（工程师配置工具 vs 数据库实体分组的区别）
- 导航深度是否合理（一般工程工具不超过 3 层）
- 关联实体的配置是否在同一视图内可访问

**3b. 视觉一致性**
- 同类型操作是否使用一致的视觉处理
- 主次操作的视觉权重是否明确
- 数据密集区（表格、曲线、代码块）的信息密度是否适当

**3c. 反馈与状态**
- 加载状态、空状态、错误状态是否都有明确处理
- 保存中 / 生成中的操作进度是否可感知
- 多步骤配置（如 Area → Requirement → Policy → Fan 的关联链）是否有完成度指示

**3d. 工程工具特定检查**
- 魔法值（如 `32768`）是否在 UI 层转换为人类可读文本
- 枚举整数（如 `pid_control_mode: 1`）是否有标签映射展示
- 技术字段名（如 `monitoring_status`、`hardware_pwm`）是否有中文说明或工具提示
- 数组类型字段（如 `temperature_range_low`）的编辑体验是否友好

**真实案例发现 — cooling-web-config 设计问题：**
```
❌ CSR 输出面板 max-height: 160px — 工程师查看生成代码的主要输出区却极为局促
❌ 曲线表格 max-height: 200px — 27 行温度-转速数据在 200px 内滚动，难以全局观察
❌ YAML 编辑器与表单各占 50% 宽度 — 实际使用中用户倾向于只用其中一种，比例应可调
⚠️ 表单网格 grid-template-columns: 1fr 1fr — 在小屏幕下字段过窄，数字输入框宽度不足
⚠️ 「生成CSR」结果直接覆盖输出区 — 没有历史记录，无法对比前后差异
```

**输出格式：**
```markdown
## 问题清单
| # | 维度 | 位置/组件 | 问题描述 | 优先级 | 建议方案 |
|---|------|----------|---------|--------|---------|

## 设计亮点
[记录现有实现中值得保留的设计决策]
```

### 第 4 步 — 设计机会点分析

在审查基础上，主动识别可以提升体验的方向，写入 `design-opportunities.md`。

**分析框架：**

**4a. 效率提升机会**
- 哪些多步操作可以合并或简化
- 哪些重复性配置可以支持批量操作或模板复用
- 哪些字段可以用智能默认值或联动推断减少用户输入

**4b. 认知负担降低机会**
- 哪些技术术语可以用业务语言替换（对操作人员更友好）
- 哪些复杂的配置关系可以用可视化方式呈现（如关系图、依赖树）
- 哪些错误可以提前在输入时预防而非提交后报错

**4c. 工程协作机会**
- 哪些设计改动可以在前端独立实现（不改后端）
- 哪些改动需要后端配合（增加字段、调整数据结构）

**真实案例机会点 — cooling-web-config：**
```
🟢 前端可独立实现：
① 魔法值转换：32768 → 「—」；pid_control_mode 1/2/3/4 → 带图示的选项卡
② CSR 输出面板可调高度（拖拽 resize）
③ 三种调速模式（节能/高性能/低噪）在同一曲线图上叠加对比显示
④ 配置完成度指示：Area → Requirement → Policy → Fan 关联链的步骤进度条
⑤ Excel 导入入口集成到 Web 页面（目前仅在 VSCode 插件中）

🟡 需后端配合：
⑥ 为枚举字段增加 label 映射（避免前端硬编码中文标签）
⑦ 配置版本历史（需后端存储支持）

🔴 产品层决策（需设计师 + PM 对齐）：
⑧ PID 控制模式的文字说明是否足够，还是需要实测数据支撑选择
⑨ 多机型配置共享（fan_board_num > 1）的 UI 如何区分槽位
```

**输出格式：**
```markdown
## 机会点清单
| # | 机会点 | 当前问题 | 建议方向 | 实现难度 | 预期收益 | 是否需要后端配合 |
|---|--------|---------|---------|---------|---------|----------------|

## 优先级建议
[结合难度和收益，给出本轮迭代应优先实现的 Top 3]
```

### 第 5 步 — 生成工程级前端代码

此步骤产出可直接接入开发工程的代码，与 hifi-to-demo 的演示代码有本质区别。

**工程级代码的标准：**
- 组件拆分符合工程实践（单一职责、可复用）
- Props 类型定义完整，字段名与后端数据模型保持完全一致
- 不使用 Mock 数据，通过 Props 或 API 接收真实数据
- 样式方案与目标工程一致（CSS 框架、变量命名）

## 来自 openUBMC Studio 的实践笔记

**从配置读业务，不只是读字段类型。** `fan_id`、`area_id`、`policy_idx` 这些 ID 背后是整机的调速拓扑关系。读懂它们的关联方式，才能设计出合理的引用选择器 UI。

**魔法值是 UI 层的责任。** `32768` 表示「无效温度」是后端内部约定，UI 绝不能直接展示。这类转换应该集中在 `formatters.js` 里统一处理，而不是散落在各组件中。

**Canvas 组件必须有表格替代方案。** 曲线编辑器用 Canvas 绘制很直观，但 Canvas 内容对辅助技术完全不透明。cooling-web-config 已经提供了曲线表格视图作为备选，这是正确的方向——设计时应确保两种视图完全等价，而非表格只是只读展示。

**工程工具的「信息密度」标准与消费者应用不同。** 专业用户可以处理更高密度的信息，但这不意味着可以用 `font-size: 11px` 或 `max-height: 160px` 的输出区。长时间使用的工具对可读性的要求更高，不是更低。

**可访问性问题根源在后端时，要写清楚。** `pid_control_mode: 1/2/3/4` 这类整数枚举，如果后端不提供 label 映射 API，前端就必须硬编码中文标签——这会导致国际化困难、标签和后端不同步的维护问题。在 accessibility-audit.md 中明确标注，推动后端改进数据结构。

**接入说明写给不了解这段代码历史的人。** 假设读者是刚加入团队的开发，不知道 `32768` 是什么、为什么 `monitoring_status` 是 CSR 同步语法字符串。写清楚这些背景，才是真正有用的接入文档。

# 后端代码 → 产品文档

这一步最容易被省略，但它是后续所有设计决策的基础。
代码里藏着产品的真实逻辑——业务规则、数据约束、实体关系——
设计师读懂它，才能做有根据的判断，而不是靠猜。

---

## 需求分析

### 用户与痛点

| 角色 | 典型场景 | 核心痛点 |
|------|---------|---------|
| 体验设计师 | 后端代码先于设计稿完成，需要快速理解「这个工具是做什么的」 | 看到一堆字段名和整数编码，不知道哪些和用户有关、哪些是内部机制；想做设计审查但没有业务上下文，审查结论流于表面 |
| 产品经理 | 接手无文档的存量工具，需要向新成员解释产品逻辑 | 只能靠运行软件来猜功能，边界不清，容易遗漏约束条件 |
| 前端开发 | 接手陌生代码仓，需要快速建立业务理解后再开发 | 字段命名用工程术语（`cmd_word: 0x30440100`），不知道这个值的含义；修改一处不知道会不会影响其他模块 |
| 新加入的团队成员 | 没有老员工传递背景，需要自学理解系统 | 文档散落在代码注释和口口相传中，找不到全貌 |

**设计师的深层痛点**：
工程工具类产品（固件配置、硬件调试、工业控制）的代码里充满了设计师不熟悉的技术概念——十六进制命令字、位字段、通信拓扑、表达式语法。这些不是障碍，是信息富矿。关键在于建立一套从「技术表达」翻译到「设计判断」的方法。

### 场景

1. **后端先行、无文档交付**：工程师完成了实现，但没有产品说明。设计师以代码为一手资料，反向建立对产品的理解，再做设计介入。
2. **新增视图的场景分析**：产品迭代时新增了若干界面，需要逐一分析：这个视图面向谁、用户在什么情境下打开它、解决什么问题、当前实现有哪些设计机会点。
3. **设计审查的上下文准备**：在跑 `/design-critique` 或 `/accessibility-review` 之前，先用此 skill 建立产品上下文，让后续审查结论有业务依据而非泛泛而谈。
4. **技术编码的设计语言翻译**：把工程师的表达（`pid_control_mode: 1`、`cmd_word: 0x30440100`）翻译成用户可理解的语言，建立设计-开发共同词汇表。
5. **存量系统设计改进**：改进现有工具的体验，需要先读懂它的数据结构和业务逻辑，才能判断哪些改动安全、哪些有风险。

### 设计点

- **翻译是设计工作**：把 `pid_control_mode: 1/2/3/4` 翻译成「慢升快降（省电优先）」，这个过程本身就是设计判断——选择什么语言、粒度多细、用什么隐喻。
- **产品文档决定审查质量**：带着「核心用户是在嘈杂机房里长时间使用的固件工程师」这个上下文去做审查，结论的精准度会高出 10 倍。
- **机会点分级决定优先级**：分清「前端独立实现」「需后端配合」「需产品层决策」，让改进建议可落地而不是空谈。

## 工作流

### Step 1 — 技术栈扫描（约 5 分钟）

读取：`package.json` / `requirements.txt` / `README` / `index.html`

确认：
- 语言与框架（原生 JS / Vue / React / Python...）
- 数据格式（YAML / JSON / XML / 二进制）
- 构建与部署方式（有无构建工具、是否支持 CDN 直接引入）
- 前端关心的核心依赖（图表库、YAML 解析器、Canvas 库、通信库）

> **openUBMC 案例**：`cooling-web-config` 是纯 HTML + CSS + JS，用 `js-yaml` CDN 解析 YAML，Canvas 绘制曲线编辑器，无需构建工具——这意味着所有工程级代码也必须保持同样的轻量约束。

### Step 1.5 — 功能模块识别与边界确认

扫描源码目录结构，按功能边界聚类，识别独立的功能模块。判断依据：
- 有独立的子目录或文件前缀命名
- 有自己的 WebView / Provider / 数据类型定义
- 使用场景完全不同，即便入口相邻或共享部分代码

区分「独立功能模块」与「同一模块的子功能」——子功能共享数据模型或操作上下文，归并到父模块里分析，不单独展开。

**输出模块清单，在继续之前发给用户确认**：

```
| 模块 | 一句话描述 | 是否独立分析 |
|------|-----------|------------|
| XXX  | ...       | ✅ 是      |
| YYY  | ...       | ❓ 待确认  |
```

如果遇到以下情况，**等用户确认再继续**：
- 两个功能共享部分代码但使用场景明显不同（入口相邻不等于是同一功能）
- 某个模块规模很小，不确定是否值得独立展开
- 某个功能被版本历史或注释标注为「已移除」「实验性」或「WIP」

确认之后，只对用户明确要分析的模块进入后续步骤，其余模块不展开。

### Step 2 — 数据模型分析（核心步骤）

读取：数据模型文档、样本数据文件、实体定义代码

**2a. 实体清单**
列出所有业务实体，每个记录：
- 实体名称（英文原名 + 中文业务含义）
- 核心字段（名称、类型、是否必填、取值范围）

→ 1: 慢升快降（省电优先）

**2d. 动态字段标注**

### Step 2+ — 陌生技术概念的识别与解释

读代码时，你会遇到各种「看不懂」的东西——一个奇怪格式的数字、一段自定义语法、一个专业术语缩写。这不是障碍，是信息密度最高的地方。

**判断一个技术概念是否需要解释的标准**：
> 如果一个刚入职的体验设计师/开发新员工看到这个词/值/格式，会不会有疑问？

会 → 需要在产品文档里解释。
不会 → 可以直接使用，不用额外说明。

---

#### 如何解释任何陌生技术概念：四步法

无论遇到什么陌生技术概念——特殊的数字格式、领域专用语法、硬件协议名称、缩写术语——都用这四个问题来解释清楚：

**① 它是什么？**（用一句话，不用行话）
把这个概念还原成最朴素的描述。不要用技术定义，用「这是一种……用来……的东西」的句式。

**② 为什么要这样设计？**（工程师的动机）
工程师不会无缘无故用一个奇怪的格式。找到他们这样做的原因——通常是为了节省空间、提高处理效率、或者遵循某个硬件规范。理解了动机，才能理解边界。

**③ 它具体是怎么工作的？**（用实例 + 图示说明）
只靠文字解释抽象概念很难被记住。用一个具体的真实例子走一遍，必要时画出结构图。「走一遍」比「讲原理」有效得多。

**④ 对用户/设计意味着什么？**（翻译回界面语言）
技术概念解释完了，最终要回答的问题是：这个概念在界面上应该怎么呈现？用户在哪一步会接触它？它对交互设计有什么约束或机会？

---

#### 什么时候用图示，什么时候用文字

| 情况 | 推荐方式 |
|------|---------|
| 概念有明确的结构（位分段、层级关系、网络拓扑） | 画结构图（ASCII 图或表格）|
| 概念涉及时序或流程（操作步骤、状态变化） | 画流程图或时序图 |
| 概念是一个类比（「就像……一样」） | 文字类比 + 一个对比例子 |
| 概念是枚举值（选项 A/B/C 各是什么意思） | 映射表格 |
| 概念是纯数值关系（公式、换算） | 带注释的计算示例 |

**原则：** 如果你读了自己写的解释，还是觉得模糊，就加一个图。图的目的不是美观，是消除歧义。

#### 三类高频出现的技术概念（附解释示范）

以下三类概念在工程工具类产品里极为常见。用它们作为「四步法」的练习示范。

---

**类型一：编码整数（一个数字里藏了多段信息）**

工程代码里常出现一个看起来像乱码的数字，比如 `0x30440100`。它实际上是把多段信息「压缩」进了一个整数。

*① 它是什么？*
一种把多个字段打包进同一个数字的存储方式。就像一个压缩文件——外面看是一个文件，解压后里面有很多东西。

*② 为什么这样设计？*
硬件通信时每次只能发送固定长度的数据（比如 4 个字节 = 32 位）。工程师把多个字段塞进这 32 位，既满足了协议要求，又一次性传递了所有必要信息。

*③ 它具体怎么工作？*（需要画图解释）

用十六进制写一个 32 位整数，正好是 8 个字符（`0x30440100`）。这不是巧合：

```
十六进制和二进制的对应关系：
每 1 位十六进制字符 = 4 位二进制（称为一个"nibble"）
8 位十六进制 × 4 = 32 位二进制

0x  3    0    4    4    0    1    0    0
   ↓    ↓    ↓    ↓    ↓    ↓    ↓    ↓
  0011 0000 0100 0100 0000 0001 0000 0000
  └─────────┘ └─────────┘ └─────────┘ ...
   第一个字段    第二个字段   第三个字段
```

工程师把字段边界设计在 nibble 边界上，这样用肉眼看十六进制字符就能分辨各字段——这是选择十六进制表示的根本原因。

字母的含义（十六进制用 0–9 和 A–F 表示 0–15）：
```
A=10  B=11  C=12  D=13  E=14  F=15
0xC = 12（十进制）= 1100（二进制）
```

同一个数里有时混用十六进制和十进制，原因是：有些字段代表「编码/地址」（用十六进制，因为位结构有意义），有些字段代表「数量/序号」（用十进制，因为数字本身有意义）。

用户如何获得这些字段定义？**不是从数字本身推导**，而是从外部规格文档（芯片手册、寄存器表、协议文档）查到的——文档里会写「第 24–31 位是功能码，第 16–23 位是命令码」，用户按照这个定义来填写或解读数字。

*④ 对设计意味着什么？*
- 用户不应该手动填写这个整数，UI 应该把各字段拆成独立输入框
- 用户填好各字段后，UI 在后台合并生成完整的编码值
- 需要显示「当前命令字预览」，让用户确认生成的值符合预期
- 字段之间可能有约束（某字段的值会影响另一字段的可选范围），需要联动校验

---

**类型二：领域表达式语言（产品自定义的迷你语法）**

代码里出现一种「不像普通代码也不像自然语言」的语法，比如 `ReadRegister(channel=2, offset=0x0C, length=4)` 或者 `<=/Scanner_Lm75_Inlet.Value`。这是产品自己定义的一套「迷你语言」。

*① 它是什么？*
一种专门为这个产品设计的指令语法，介于「人话」和「编程语言」之间。用于描述计算逻辑、查询条件或操作规则，不需要通用编程知识就能（理论上）看懂。

*② 为什么这样设计？*
让不会写代码的用户（运维人员、工程师）也能描述「我想做什么操作」，而不需要学习一门完整的编程语言。

*③ 它具体怎么工作？*（关键是区分不同的使用模式）

同一套表达式语法，往往有两种截然不同的使用方式：

```
调试模式（即席执行）          用例模式（脚本化复用）
─────────────────────         ──────────────────────
用户意图：「我现在想试         用户意图：「我要定义一套
试这条指令是否有效」           可反复执行的标准操作」

→ 填一条表达式                → 填多条表达式，组成序列
→ 立刻执行，看结果             → 保存为「用例」，按需触发
→ 不保存                      → 可共享给团队

类比：计算器 vs 宏录制
```

输入参数的作用：
```
不带参数：ReadRegister(channel=2, offset=0x0C)  ← 只能读这一个寄存器
带参数：  ReadRegister(channel=$ch, offset=$off) ← 模板，可复用

添加多个参数值（批量执行）：
ch = [0, 1, 2, 3]，offset = 0x0C
→ 自动执行 4 次，分别读取 channel 0/1/2/3 的同一个寄存器
→ 相当于「一个表达式，多组输入，对比结果」
```

*④ 对设计意味着什么？*
- 调试模式和用例模式是两种不同的使用意图，不能共用同一个表单加一个开关了事
- 切换模式时，用户已填写的内容不能丢失
- 参数输入需要类型提示和值域说明（`$ch` 应该是整数，取值 0–7）
- 批量执行的多组结果适合表格对比展示，不适合逐条显示

**类型三：通信拓扑（命令发给谁）**

代码里有各种设备地址（`0x48`、`0x2C`）和总线编号（`i2c_bus: 1`），还有「发送命令」「广播」之类的操作。这描述的是一个硬件通信网络。

*① 它是什么？*
多个硬件设备连接在一条共享线路上，每个设备有唯一「门牌号」（地址）。发送命令时需要指定门牌号，命令才能准确送达目标设备，不影响其他设备。

*② 为什么这样设计？*
硬件资源有限，不能为每对设备之间都拉一条专用线。共享总线 + 地址寻址，是一种经济实用的多设备通信方案。I2C、SPI、CAN 等都是这类协议，原理相同，细节不同。

*③ 它具体怎么工作？*

```
通信主机（发令者）
     │
     ├── 总线 0 ────┬── 地址 0x48：进风口温度传感器
     │              └── 地址 0x4A：排风口温度传感器
     │
     └── 总线 1 ────┬── 地址 0x2C：风扇控制器
                    └── 地址 0x58：电源管理芯片

发送命令 → 指定「总线 + 地址」→ 只有那个设备响应
广播命令 → 不指定地址（地址 0x00）→ 总线上所有设备都响应
```

命令字和通信拓扑的关系：命令字里通常有一个字段是「目标设备地址」，这个地址指向拓扑图里的某个节点。命令字决定「做什么」，拓扑决定「发给谁」。

*④ 对设计意味着什么？*
- 用户选择「发给谁」时，应该能看到设备列表或拓扑图，而不是手动填写地址数字
- 发送前应显示「将发送给：[设备名称]（地址 0x48，总线 0）」，让用户确认目标
- 广播操作影响所有设备，必须有二次确认警告
- 拓扑结构如果发生变化（新增设备），UI 应该能反映，不依赖用户手动更新地址

#### 遇到其他陌生技术概念时的处理原则

不是每个项目都有命令字或通信拓扑，但每个工程工具类产品都有它自己的「术语墙」。遇到任何看不懂的概念时：

1. **先问「这解决了什么问题」**——工程师不会无缘无故发明一个复杂概念，找到动机是理解的捷径。
2. **找最小可理解的例子**——不要试图先理解全貌，找一个最简单的真实值（如 `0x30440100`），走一遍完整的解析过程。
3. **用类比建立直觉**——「就像……」的类比往往比技术定义有效 10 倍。好类比的标准：听的人说「哦，就是那个意思啊」。
4. **画出来**——任何有结构的概念（层级、分段、网络），画出来之后通常立刻清晰。不需要漂亮，ASCII 图已经够用。
5. **最后问「所以呢」**——翻译回界面语言：这个概念对用户操作有什么影响？界面上应该怎么处理？

| 3 | **使用场景** | 2–4 个具体场景，用叙事描述：用户在做什么 → 遇到什么 → 怎么用 → 得到什么 |
| 4 | **目标用户** | 角色、使用频率、使用环境、核心诉求、典型触发点 |
| 5 | **痛点分析** | 当前没有工具时的操作成本（量化）；每个痛点段落结尾用固定格式提炼：「**痛点核心：**……」 |
| 6 | **设计机会点** | 🟢 前端可独立实现 / 🟡 需后端配合 / 🔴 需产品层决策 |
| 7 | **竞品分析 + 专利潜力** | 见下方格式要求 |

**第 7 项：竞品分析 + 专利潜力评估**

竞品分析的重点是找「差距」，而不是罗列功能：

```
| 竞品 / 行业方案 | 具备的能力 | 与本功能的差距 |
|----------------|-----------|--------------|
```

> 差距是设计决策的依据——「我们能做而别人没做」才是机会，「别人做了一样的」只是参考。

专利潜力评估回答三个问题：
1. 这个功能组合（操作方式 × 界面交互模式 × 应用场景）在行业内是否有先例？
2. 差异化点是哪几个关键词的组合（注意：是**组合**，而非单个关键词）？
3. 专利潜力评级：**高**（组合未见先例）/ **中**（组合新颖但单项不稀缺）/ **低**（已有公开先例）

**专利描述原则**：专利点只从**用户交互操作角度**描述——触发方式、界面交互模式、状态连续性设计等。不引用前端实现技术（WebView、localStorage、API 名称等）。保护对象是用户体验的创新交互流程，而非技术实现。

```
✗ 错误描述：「使用 localStorage 缓存表达式历史」
✓ 正确描述：「用户完成计算后，历史记录自动出现在下次打开时，无需手动保存」
```

### Step 5 — 设计机会点汇总（三列流视图）

将所有模块的设计机会点汇总为一张**三列流视图**，表达「痛点 → 洞察 → 策略」的因果链，而不是平铺的机会点列表。

**三列结构**

| 列 | 内容 | 写法要求 |
|----|------|---------|
| 第一列 | **用户痛点** | 具体、可量化，带操作成本（来自各模块的痛点分析） |
| 第二列 | **洞察汇聚** | 从多个痛点中提炼的深层原因，通常比痛点数量少（多对一） |
| 第三列 | **设计策略** | 对应洞察的具体设计方向，标注 🟢/🟡/🔴 实现层级 |

三列之间用连线标注「哪些痛点汇聚成哪条洞察，哪条洞察导出哪个策略」。

**洞察汇聚的写法**：不是痛点的复述，而是提炼「为什么这些痛点会同时出现」的深层原因。

```

```

在 Step 8 的汇总幻灯片里，这张三列流视图以 HTML 可视化渲染（连线 + 色块），不是表格。

**SVG 连线（三列流视图）技术约束**

使用 `CSS transform:scale()` 实现 16:9 缩放时，`getBoundingClientRect()` 返回的是屏幕像素坐标而非布局坐标，导致 SVG 坐标系与元素位置不匹配、连线错位。

修复方式：从 `deck.style.transform` 提取当前 scale 值，将所有 rect 坐标除以 scale 后再传入 SVG path：

```js
const scaleMatch = deck.style.transform.match(/scale\(([\d.]+)\)/);
const scale = scaleMatch ? parseFloat(scaleMatch[1]) : 1;
const rect = el.getBoundingClientRect();
const x = rect.left / scale;
const y = rect.top / scale;
```

## 来自 openUBMC Studio 的实践笔记

**先读数据模型，再读代码。** `data-model.md` 里的信息密度比源代码高得多——它已经做了一次结构化，你从这里开始能节省大量时间。

**样本数据是最好的字段说明书。** `sample-cooling.yaml` 里的真实值告诉你字段的实际值域，比类型定义（`string`）更有用。

**魔法值是最优先要找的。** `32768`、`-1`、`"N/A"` 这类特殊值往往隐藏在业务逻辑里，不找就会漏到 UI 上——这是最容易被用户骂的那类 bug。

**十六进制不是障碍，是结构信号。** 看到 `0x30440100` 不要跳过——这个写法告诉你这个值被分成了 4 段，每段 8 位，每段有独立含义。设计师的工作是把这些「段」翻译成用户能填写和理解的字段。

**位字段的来源永远是外部文档，不是命令码本身。** 用户不是「拿到命令码来解析字段」，而是「先有规格文档知道各段含义，再按规格填写命令码」。UI 的目标是让规格文档里的字段定义变成用户的操作界面，而不是让用户手动做位运算。

**调试模式和用例模式的根本差异是使用意图，不是功能差异。** 调试模式是「我要试试这条命令」，用例模式是「我要定义一套可重复的操作」。两者的 UI 结构应该反映这种意图差异，而不是共用同一个表单加一个 toggle。

**通信拓扑的可见性直接影响操作安全性。** 工程师发一条命令影响的是一个具体的物理设备，如果 UI 里看不出来命令发给了谁，就容易误操作影响不相关的设备。让「命令目标」在每次操作前都可见，是工程工具类产品最基本的安全设计。

**产品文档是审查工具的放大镜。** 同样的「对比度不足」，有了「核心用户是在嘈杂机房里长时间使用的固件工程师」这个上下文，严重程度和修复优先级会完全不同。
