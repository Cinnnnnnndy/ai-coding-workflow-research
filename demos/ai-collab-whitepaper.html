总 · 核心判断

AI 在协同中扮演什么角色

传统流程里，设计师在工程代码完成后才介入——审查、标注、返工。AI 协同改变的不是设计结论，而是介入的时机。

之前的问题 设计师拿到的是运行中的软件或设计稿，没有代码上下文，审查结论流于表面；工程师拿到的是标注文档，翻译成本高，还原度有限。
AI 改变的节点 设计师可以直接读代码——AI 负责翻译技术语言；工程师可以直接收到可接入的代码——AI 负责从设计意图生成结构。协同的起点前移了。
仍然不变的 设计判断、产品策略、用户洞察的提炼仍是人的工作。AI 提供的是原料和结构，而不是判断本身。

分 · 模式一

读懂代码，建立共同语言

工程工具类产品的代码里充满了设计师不熟悉的术语——十六进制命令字、位字段、通信拓扑、表达式语法。这些不是障碍，是信息密度最高的地方。

核心问题：「后端给了代码，设计师怎么知道这个工具是做什么的？」——在没有产品文档、没有老员工口传的情况下，代码是唯一的一手资料。

四步法：解释任何陌生技术概念

无论遇到什么格式的数字、什么语法的表达式、什么缩写的术语，用四个问题把它讲清楚：

① 它是什么
用一句话，不用行话
「这是一种……用来……的东西」的句式
不要用技术定义，用朴素描述
② 为什么这样
工程师的动机
节省空间 / 提高处理效率 / 遵循硬件规范
理解了动机，才能理解边界
③ 怎么工作
用实例 + 图示走一遍
拿一个真实的值，完整解析
「走一遍」比「讲原理」有效得多
④ 对设计意味着
翻译回界面语言
用户在哪一步接触它？
对交互有什么约束或机会？
这个四步框架是通用的——不只适用于命令字，任何「刚入职的设计师/开发新员工看到会有疑问」的东西都应该用这四个问题回答。
模式一 · 真实案例

案例：命令字 0x30440100 的设计翻译

SMC 偏移量计算器的核心输入是一个十六进制整数。工程师自然地写 cmd_word: 0x30440100，设计师看到这行代码时完全不知道它意味着什么。

SMC 偏移量计算器 · 代码原文
// 工程师的表达 cmd_word: 0x30440100 // 设计师看到的：一个神秘的十六进制数 ❌ 不知道它有几个字段 ❌ 不知道每段代表什么含义 ❌ 不知道UI应该让用户填什么
AI 协同后：四步法翻译结果
① 它是什么？ 一个 32 位整数，里面打包了 4 段信息 就像一个「压缩文件」，外面一个数， 解压出来是多个独立字段。 ③ 怎么工作（走一遍 0x30440100） Bit 31-24: 0x30 → 功能码（读操作） Bit 23-16: 0x44 → 设备类型（CPU 模组） Bit 15- 8: 0x01 → 功能号（温度读取） Bit 7- 0: 0x00 → 偏移量（寄存器起始位置） ④ 对设计意味着什么？ ✓ UI 应该是 4 个独立输入框，而非一个整数 ✓ 每个字段有枚举值 → 应该是下拉选择 ✓ 实时预览拼装结果 → 帮用户确认
痛点

工程师每次发命令都要查阅两份文档

功能码对照表 + 设备类型对照表分开存放，手动拼装时容易出错。

痛点核心：翻译比填写更高频——工程师花在「找对照关系」上的时间远多于「填写命令」本身。
洞察

规格文档里的字段定义 = 用户的操作界面

用户不是「拿到命令字来解析字段」，而是「先有规格文档知道各段含义，再按规格填写」。

UI 的目标是让规格文档里的字段定义变成用户的操作界面，不是让用户手动做位运算。

设计机会

结构化输入 + 实时预览

🟢 把一个整数输入框拆成 4 个结构化字段，每个字段有枚举选择和说明。

🟢 底部实时显示拼装后的命令字（十六进制 + 二进制），让用户看到「我填的东西是这样组合的」。

分 · 模式二

从代码提取设计洞察

有了产品文档之后，设计师可以做真正有根据的设计审查——不是「对比度不够」这样的泛泛观察，而是「核心用户是在嘈杂机房里长时间工作的固件工程师，这个颜色对比度不足会影响他在强光环境下的使用」。

AI 的具体作用：系统化地把代码里散落的行为约束（字段依赖关系、操作顺序约束、边界值处理）翻译成「设计审查的检查项」，把原本依赖经验的审查变成可重复执行的流程。

模式二 · 真实案例

案例：批量表达式计算器的痛点→洞察链

管道表达式语法 "<=/Scanner_Lm75_Inlet.Value" 是整个工具最难理解的部分。AI 协同把这个「技术语言的迷宫」拆成了设计决策的依据。

批量表达式计算器 · Pipe 语法解析
// 工程师写的表达式 sensor_reading: "<=/Scanner_Lm75_Inlet.Value" ① 它是什么？ 一种「实时读取传感器值」的简写语法 <= 管道符，代表「从……读取」 /Scanner_Lm75_Inlet 传感器名称（在系统拓扑里的路径） .Value 读取它的当前值 ② 为什么这样设计？ 避免在配置文件里硬编码传感器数值，让配置文件「动起来」 ③ 调试模式 vs 用例模式的本质差异 调试模式：「我要试试这条命令，看看结果」 → 单次执行、看返回值 用例模式：「我要定义一套可重复的操作流程」 → 批量执行、记录历史 ④ 对设计意味着什么？ 两种模式的 UI 结构应该反映使用意图，而不是共用一个表单加一个 toggle

痛点 → 洞察 → 设计策略

批量表达式计算器 · 三列流视图
用户痛点

P2 调试单条表达式和批量运行混在同一个界面
操作意图不明确

P1+P2 同时出现的根因是：工程师花大量时间在「理解语法结构」而不是「填写具体值」

调试（试）和用例（固化）是两种不同的认知模式，共用界面导致双方都不顺手

S1 语法引导 + 实时补全

分 · 模式三

从代码到工程级实现

设计分析完成后，下一步是把「设计意图」转换成「工程师可以直接用的代码」。这一步过去完全依赖前端开发，现在设计师可以直接参与生成。

边界说明：AI 生成的工程代码处理的是「视觉结构 + 交互逻辑」，动态数据注入和业务逻辑仍需工程师接入。设计师主导静态结构，工程师接手动态部分。

能效调速配置模板 · 完整协同五步

这个工具视图是 openUBMC Studio 最复杂的一个配置界面，涉及 8 种实体类型、跨实体引用关系、以及动态语法字段。以下是实际走过的协同过程：

Step 1
读代码
识别 8 种实体：CoolingConfig、CoolingArea、CoolingFan、CoolingPolicy…
发现核心约束：先定义 CoolingArea，才能建立 CoolingRequirement 引用关系
标注魔法值：sys_hdds_max_temperature: 32768 → 「无效」标记，UI 必须转为「—」
Step 2
产品文档
输出 product-doc.md：实体清单 + 关系层级图 + 枚举映射表
把 pid_control_mode: 1/2/3/4 翻译为：慢升快降 / 快升慢降 / 慢升慢降 / 快升快降
Step 3
设计分析
找出 5 个痛点：实体关系不可见、操作顺序不明、动态语法无提示…
提炼 9 个设计机会：关系可视化 / 操作向导 / 模板复用 / 实时预览…
WCAG 2.1 AA 可访问性审查：颜色对比度 + 键盘导航 + 表单标签
Step 4
生成代码
生成工程级前端：纯 HTML+CSS+JS，无构建工具，匹配 openUBMC Studio 技术栈
包含：实体关系可视层 + 配置表单组件 + 魔法值过滤逻辑

分 · 摩擦与边界

发现的摩擦和当前的边界

诚实地记录实践中遇到的摩擦点，不是为了否定协同价值，而是为了让下一个使用这套流程的人知道「哪里需要格外注意」。

⚠ 高保真仍是不可省的环节

01
问题 AI 生成的 Code Demo 在交互逻辑和视觉结构上达到「可用」，但距离现网高保真设计稿的细节还原仍有距离——字体精度、间距一致性、边界状态处理等。
02
现状 两条线并行：工程代码 + 高保真设计稿都需要交付。设计师负担增加，两份产出之间的一致性需要手动维护。
03
方向探索 Claude Design 对「已有基础 HTML + 产品文档」的场景效果较好——可以在 Code Demo 基础上做视觉优化，减少高保真和代码的偏差。但这条路尚在探索中。

✕ Design 与 Code 的协同割裂

01
问题 Claude Design 和 Claude Code 是两个独立工具。在 Claude Code 里完成分析和代码生成后，无法直接唤起 Claude Design——需要手动跳转，上下文不能自动传递。
02
影响 视觉优化环节的成本没有降低；Code 生成的文件结构和 Design 的输入格式有时不完全匹配，需要手动整理才能传递。
03
临时方案 先用 /code-to-product-doc 生成产品文档，把文档作为 Claude Design 的输入上下文，再结合基础 HTML 做视觉优化。这是目前最顺畅的衔接方式。

哪些判断仍然是人的工作

产品策略判断

「这个功能值不值得做」「哪个痛点优先级更高」「竞品差距是机会还是陷阱」——这些需要产品全景和业务上下文，AI 提供原料，人做判断。

用户洞察提炼

从多个痛点提炼洞察（比如「翻译比填写更高频」），不是单纯归纳，是理解为什么这些痛点会同时出现的深层原因。这个提炼过程需要设计师的判断力。

新型交互模式

对于行业里没有先例的交互设计（专利潜力高的机会点），AI 能提供参考方向，但最终的交互设计决策需要设计师自己做。

总 · 能沉淀的资产

从实践到可复用方法

这次探索不是一次性的项目，而是在积累「可以一直用下去的方法资产」。三个月的实践，沉淀下来最有价值的是这几件事：

Skill 资产 三个 Claude Code Skill 是最直接的沉淀——安装后任何项目都能用，触发词让 AI 自动进入正确的工作流，不需要每次重新写提示词。
方法论资产 四步法（是什么→为什么→怎么工作→对设计意味着）是通用框架，适用于任何工程工具类产品里的陌生技术概念。

cmd_word 32 位编码整数，将功能码、设备类型、功能号、偏移量四段信息打包存储。UI 层不应让用户直接填写整数，应拆成结构化字段。 模式一 · SMC 案例
位字段（Bit Field） 将一个整数按位分段，每段有独立含义。0x30440100 = 功能码 0x30 | 设备类型 0x44 | 功能号 0x01 | 偏移量 0x00。 模式一
Pipe 语法 BMC 批量表达式计算器使用的动态语法。<=/Scanner_Lm75_Inlet.Value 表示「从传感器实时读取数值」，不是静态字符串。 模式二 · 表达式计算器
魔法值（Magic Value） 用特殊数字表示特殊状态的约定，如 32768 代表「无效/获取不到」。UI 必须拦截转换，不能直接展示给用户。 模式三 · 能效配置
