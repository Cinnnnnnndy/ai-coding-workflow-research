技术概念解释 — 四步法

凡是「刚入职的体验设计师/开发新员工看到会有疑问」的技术概念，都用四步法解释：

① 是什么
用一句话，不用行话，「这是一种……用来……的东西」
② 为什么
工程师这样设计的动机——节省空间 / 提高效率 / 硬件规范
③ 怎么工作
用一个真实的值完整走一遍，必要时绘制结构图
④ 设计含义
翻译回界面语言：用户在哪一步接触它，UI 应该怎么处理

主要产出

读代码 · 真实案例

案例：SMC 偏移量计算器 — 命令字 0x30440100

// 工程师的原始代码
cmd_word: 0x30440100

// 设计师拿到这行代码时的状态：
// ❌ 不知道这个数有几个字段
// ❌ 不知道每段代表什么含义
// ❌ 不知道 UI 应该让用户填什么

四步法应用结果

① 是什么
32 位整数，里面打包了 4 段信息——就像压缩文件，外面一个数，解压出来是多个独立字段
② 为什么
BMC 硬件寄存器地址空间有限，多字段打包成一个整数传输效率更高，也与硬件规范对齐

③ 怎么工作 — 0x30440100 完整解析

0x30
Bit 31-24
功能码
读操作
0x44
Bit 23-16
设备类型
CPU 模组
0x01
Bit 15-8
功能号
温度读取
0x00
Bit 7-0
偏移量
寄存器起始

④ 对设计意味着什么

之前：一个整数输入框，用户手动填写 0x30440100，需要自己查两份对照表拼装
机会：4 个结构化下拉选择框，每个字段有枚举选项和说明；底部实时预览拼装后的命令字
痛点核心：翻译比填写更高频——工程师花在「查找对照关系」上的时间远多于「填写命令」本身

设计分析 · 真实案例

案例：批量表达式计算器 — 痛点 → 洞察 → 策略

用户痛点
P1

不知道 pipe 语法怎么写

每次使用前需查阅两份文档，约 5–8 分钟/次；表达式路径需手动拼接传感器名称

P2

调试与用例操作意图混乱

「试一下这条命令」和「定义一套可重复流程」共用同一个输入界面，切换时无明确的状态区分

P3

执行结果无历史记录

结果只显示在当前页面，刷新即消失；工程师依赖截图保存，无法回溯对比

→
洞察汇聚
L1

翻译比填写更高频

P1 + P2 的根因：工程师花在「理解和查找语法结构」上的时间，远多于「填写具体参数值」本身。工具把认知负担放错了位置。

L2

操作意图决定界面结构

调试（单次探索）和用例（批量固化）是两种不同的认知模式，界面应分别服务两种意图，而不是加一个 toggle 勉强兼容。

→
设计策略
S1

语法引导 + 路径补全 🟢 前端独立

输入框内置传感器树形选择，自动生成正确的 pipe 路径；输入时高亮语法段落，识别错误立即提示。

S2

调试 / 用例模式结构分离 🟡 需后端

两个独立的交互入口，调试模式轻量即时，用例模式支持命名保存和批量复用，历史记录自动保留。

补充信息

竞品分析 — 差距方向

方案具备能力与本功能的差距
通用 Postman 风格批量请求、历史记录无传感器实时值实时绑定；无 pipe 表达式引导
BMC 命令行工具完整命令覆盖纯文本，无语法引导，无结果可视化

专利潜力评估

差异化点组合：传感器实时值绑定 × pipe 语法引导 × 调试/用例意图分离

潜力评级：中 单项有先例 · 组合新颖

发现与摩擦

引入 Claude Design — 场景与摩擦

引入原因

Claude Code 在生成代码的交互逻辑方面能力强，但对视觉细节（字体精度、间距一致性、视觉层次）的优化精度偏低。引入 Claude Design 补足这一环节。

效果较好的使用场景

适用
已有基础 HTML（Code Demo 产出） 已有产品文档（/code-to-product-doc 产出）

以产品文档作为上下文输入 Claude Design，结合基础 HTML 做视觉优化——这是目前最顺畅的衔接方式。

当前工作流：Step 02 产出 product-doc.md → 输入 Claude Design → 视觉优化后的 HTML → 继续在 Claude Code 里做逻辑接入

主要摩擦

无法直连
Claude Design 和 Claude Code 是两个独立工具，需手动跳转切换
上下文断裂
Code 中的分析上下文不能自动传递给 Design，需要手动整理输入
格式不匹配
Code 生成的文件结构有时与 Design 的输入格式不完全对应，需手动整理

对比：Design 介入前后

Claude Code 直接输出经 Claude Design 优化
交互逻辑✓ 完整✓ 完整
视觉细节精度⚠ 偏低✓ 提升明显
与高保真偏差较大缩小，但仍存在
切换成本—需手动跳转 + 重新整理上下文
发现与摩擦

当前发现的主要摩擦点

①
高保真交付仍是不可省的环节
  • Code Demo 已达交互和视觉「可用」，但与现网高保真的细节还原仍有明显距离
  • 字体精度、间距系统、边界状态处理等需要高保真补全
  • 当前结论：两份产出（Code Demo + 高保真）都需要交付给前后端
  • 设计侧工作量增加，两份产出之间的一致性需手动维护
探索中：Claude Design 在视觉优化环节有改善，但还不能完全替代高保真的细节精度。
②
Design 与 Code 的协同割裂
  • 两个工具独立，不能在 Claude Code 里直接唤起 Claude Design
  • 上下文不自动传递——Code 里的分析结果需手动整理后输入 Design
  • 文件格式有时不匹配，需要额外整理步骤
  • 当前绕过方式：产品文档作为中间载体手动传递
③
模块边界判断需要人工确认
  • Step 1.5（功能模块识别）中，AI 有时难以准确区分「独立功能模块」和「同一模块的子功能」
  • 案例：SR 文件预览和 MIB 检查入口相邻，但是完全独立的功能单元
  • 需要设计师确认模块边界，再进入逐模块分析
Skill 里已内置「模块边界确认」步骤——识别后先向用户列出模块清单，确认后再继续。
