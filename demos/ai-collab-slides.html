--- /code-to-product-doc · SKILL.md ---
## 用途
把工程代码翻译成设计师能直接用的语言。
产品文档是设计介入「后端先行」项目的地基——为后续设计审查、A11y 分析、工程实现提供统一上下文。
## 典型场景（5 个）
1. 后端先行、无文档交付 → 设计师以代码为一手资料，反向建立产品理解
2. 新增视图的场景分析 → 这个视图面向谁、解决什么问题、有哪些机会点
3. 设计审查的上下文准备 → 跑 /design-critique 或 /accessibility-review 前先用此 skill
4. 技术编码的设计语言翻译 → pid_control_mode:1 → 「慢升快降（省电优先）」
5. 存量系统设计改进 → 判断哪些改动安全、哪些有风险
## 工作流（5 步）
Step 1 — 技术栈扫描
读 package.json / README → 确认语言/框架/数据格式/核心依赖
Step 1.5 — 功能模块识别与边界确认
扫描源码目录，聚类功能模块，输出模块清单发给用户确认后再继续
Step 2 — 数据模型分析（核心）
识别所有业务实体 → 分析字段含义与约束 → 建立技术术语↔业务含义映射
Step 3 — 业务逻辑梳理 + 机会点识别
分析核心流程 → 梳理用户操作路径 → 分级机会点（前端独立/需后端/需产品层）
Step 4 — 产出 product-doc.md
按标准结构输出；可选：data-dictionary.md（工程命名↔业务语言↔UI展示）
## 产出规范
product-doc.md
功能定义 · 用户场景 · 数据结构 · 核心流程 · 设计机会点 · 注意事项
→ 供设计审查、A11y 分析、团队对齐、新成员 onboarding 使用
data-dictionary.md（可选）
工程命名 ↔ 业务语言 ↔ UI 展示对照
→ 消除命名歧义，供前后端长期查阅
## 注意事项
- 代码可读性极差或缺少类型/注释时，先与工程师对齐
- 跨模块超大项目，按模块分次运行，避免上下文过载
- 本 skill 不替代用户研究，focus 是理解现有实现而非从零设计

## 工作流（6 步）
Step 1 — 接收后端交付物
确认已有资料（代码/API/YAML/README），确认技术栈与数据格式
Step 2 — 读代码 → 产品文档 （→ /code-to-product-doc 独立 skill）
产出：功能定义 · 用户场景 · 数据结构 · 核心流程 · 设计机会点 · 注意事项
⚠ 最容易被省略但最不该省略的一步，是后续所有分析的基础
Step 3 — 可访问性审查 → accessibility-audit.md
WCAG 维度逐项评估；根源在后端时（如枚举缺 label）明确标注以推动后端改进
Step 4 — 设计检查 + 机会点 → design-review.md + design-opportunities.md
检查：一致性/层级/可读性/信息架构；机会点分三级（前端独立/需后端/需产品层）
P-L-S 框架：Pain 痛点 → Learning 洞察 → Strategy 策略
Step 5 — 工程级代码 → frontend/
字段名对齐后端数据模型；含 loading/empty/error 三态；可直接接入工程
Step 6 — 接入文档 → integration-guide.md
引入方式、依赖说明、联调检查项
## 产出规范
code-reading.md
后端交付物解读：数据结构 · 业务逻辑 · 接口契约（是所有后续分析的基础）
accessibility-audit.md
WCAG 维度 + 工程侧建议；根源在后端的问题单独标注
design-review.md
设计检查：一致性 · 层级 · 可读性 · 信息架构问题清单
design-opportunities.md
设计机会点分析 + 优先级建议（分三级：前端独立/需后端配合/需产品层决策）
frontend/ （工程级代码）
字段名对齐后端，含完整三态处理，可直接接入工程（≠ Demo）
→ 区别于 /hifi-to-demo：起点是后端代码，字段严格对齐，含错误/空/加载态
integration-guide.md
接入说明：引入方式 · 依赖说明 · 联调检查项
## 注意事项
- 依赖前一步产品文档质量——文档不足时设计分析会悬空
- 不替代视觉设计——focus 是结构、可访问性、状态、机会点
- 纯新功能从零设计时不适用——本 skill 是「反向分析」，需已有代码
- A11y 问题根源在后端时（如整数枚举缺 label），要明确标注推动后端改进

## Phase 1 工作流 — 只出文档
Step 1 — 收集输入

⚠ 写任何代码之前先完成此步
Step 3 — 交互分析 → interaction-spec.md
状态枚举 · 交互序列 · 待设计师确认清单
Step 4 — 接口&业务逻辑分析 → interface-contract.md
接口定义 · 字段→UI状态映射 · 业务规则 · 前后端责任划分 · 联调待确认
Step 5 — 技术选型 → tech-selection.md

⏸ 节点 A — 文档 + 高保真二次输入，触发 Demo 生成
## Demo 生成工作流
Step 6 — 交叉校验文档与高保真
用高保真核验文档偏差，修正后再生成代码
Step 7 — 生成 demo/index.html
严格基于已校验文档 · Mock 字段=接口契约 · 业务规则参照前后端边界
Step 8 — 易用性审查 → quality-review.md
视觉精度 + 可读性 + 交互反馈 + 空/错误状态
⏸ 节点 B — Demo 评审确认 → Phase 2
## 7 个交付物

L1 → L2 跃迁

近期 · 必须打通
⚠ 5 大阻力 · 为什么还在 L1
· 代码可维护性：AI 生成质量参差，未达「可合入主干」标准
· 团队能力差距：Vue/React + Git + Mock 数据设计
· 组织评价缺位：评价仍按「出图」，写代码无认可路径
· 提示词资产分散：仅单页面 prompt，缺组件级 prompt 库
· 前端工程师角色模糊：设计师全量交付代码 → 前端沦为「接收/联调」
需澄清：设计交付『高保真 UI 组件 + Mock 契约』，前端保留 src/services、路由、全局状态、CHANGELOG
✓ 3 步路径 · 12–18 个月
· 0–3 月：扩展 skill + 组件级 prompt 库 + 选 1 模块试点
· 3–9 月：Vue/React 培训 + Git 工作流 + 代码评审流程
· 9–18 月：推广 3+ 项目 + 新评价指标落地

L2 → L3 跃迁

远期 · 部分可行
L3-前端：可行（设计师主导前端仓 + Mock）
L3-全栈：短期不现实（后端系统层始终需要工程师）
有利因素
· CSR 数据结构固定，可构建高质量 Mock
· 可视化是设计师天然强项
· Webview 架构支持设计代码直接运行
· AI 在静态/可视化生成上能力成熟
核心障碍
· 后端复杂度（CSR/I²C）需专业工程师
· 生产代码性能/安全标准超 AI Demo
· 设计师前端架构决策风险尚未达到
· 组织评价体系尚未适配「写代码」
核心趋势：设计师工作量 30%→90%，开发 100%→25%——本质是「前端归设计，后端归工程」的职责重新划分。当前优先打通 L2，L3 视 AI 能力与组织准备度而定
协同层级 · 2/2 · 会议结论 + 使用观察

会议关键结论 · 使用过程观察

🤝 第一次设计 × 开发协同会议 · 关键结论
2026 · 首次对齐
结论 01
设计师在 AI 生成中的角色
定义设计系统 · 组件库调用 · 初步代码 Demo · 可行性分析 · 交互细化 · 落地检视。精确描述设计稿与样式，提升 AI 生成准确性。
结论 02
静态页面 vs 模块化开发
AI 可较直接生成静态页面；复杂业务逻辑仍需手动处理。静态 div 由设计侧 vibe coding 产出，开发侧注入动态数据与业务逻辑。
结论 03
设计稿 → 代码的转换路径
提供设计稿截图 + 演示代码 + 详细需求描述，三者结合可显著提升生成质量。精准描述样式是 AI 理解设计意图的核心输入。
结论 04
L0–L3 协同模式确认
当前阶段以可行性分析为主，帮助开发理解设计实现难度。前端需结合业务逻辑和接口定义，对 AI 生成代码进行优化调整。
结论 05
技术栈选择与迁移
设计侧 vibe coding Demo 提供选型参考，部分代码可复用。决策：先以 Vue 实现核心功能，同步研究 React 迁移方案。Demo 可供 AI 学习参考。
结论 06
设计师技能演进方向
提示词工程 · 设计系统定义 · 组件库调用规范 · 落地检视，成为新核心技能。传统标注文档逐步被代码 Demo + 提示词资产库替代。
