01 研究洞察

项目背景 & 研究范围

研究范围 — 三个工具视图

SMC 偏移量计算器 批量表达式计算器 能效调速配置模板

openUBMC Studio 是运行在 VS Code Webview 内的固件工程师工具集。三个工具视图均由后端工程师先完成基础实现，是最高频使用的功能模块。

起点条件 — 后端先行，无产品文档

  • 后端工程师先完成工具视图基础实现，交付可运行 HTML Demo + 数据文件（YAML/JSON）
  • 设计师以代码为一手资料反向理解产品逻辑
  • 没有产品经理同步出需求文档，没有设计规范参考
  • 技术概念密集：十六进制命令字、管道表达式、I²C 拓扑、CSR 对象层级
研究挑战：设计师在理解产品之前，需要先跨越「读懂工程代码」这道门槛。

技术栈约束

运行环境
VS Code Extension Webview 无 Node 构建步骤
技术范围
纯 HTML · CSS · JS js-yaml CDN 解析 YAML
数据格式
SR 文件为 JSONC（含注释） 需 jsonc-parser，标准 JSON.parse 报错

AI 生成的代码必须满足同等约束——这直接影响了各 Skill 的提示词设计边界。

01 研究洞察

目标用户 & 场景研究

场景 A · 开发阶段 — 构造命令字

工程师查到芯片手册：Fn=0x06、Cmd=0x4480、RW=1，需要手算位移合成 32 位整数，填入 SR 文件的 offset 字段。手算过程约 5–10 分钟，且错误只在运行时才暴露——命令字合法但字段写错，固件不报编译错误，硬件访问异常后才能发现。

高频 · 每天错误延迟暴露
场景 B · 调试阶段 — 解读日志数字

线上报错，日志显示命令失败，偏移量是 809893888（十进制）。工程师需要临时开 Python 做「十进制→十六进制→拆字段」三步转换，才能知道这是哪个功能模块在哪个设备上的哪类操作出了问题。每次调试都需要搭脚手架，中断工作流。

按事件触发每次搭脚手架
场景 C · 代码 Review — 理解魔数

同事代码里有 smc_read(0x30280001)，Reviewer 完全不知含义，无法判断字段值是否合理。需询问原作者或查阅两份对照文档。每次 Review 产生大量隐形沟通成本，且严重依赖原始开发者仍在团队。

低频隐形沟通成本

目标用户角色

角色主要操作频率
固件工程师编码（开发期）每天
调试工程师解码（排查期）按事件
代码 Reviewer解码（审查）低频
新团队成员两个方向学习期

操作安全性要求

  • 字段填错一位可能命令发给错误设备，校验必须即时且可读
  • 超出 32 位时代码取低 32 位（>>>0）— UI 必须显示溢出警告
  • offset 为变量引用（${Var}）时计算器需优雅处理
  • 管道表达式最长 1000 字符硬限制，超长不解析需明确提示
工程工具类产品：错误代价远高于消费类产品，安全性约束是设计的第一优先级。
01 研究洞察

关键洞察汇聚 — 从场景到设计方向

用户痛点
P1

命令字手算 5–10 分钟

位宽容易记错，错误运行时才暴露

P2

日志十进制数字不透明

809893888 毫无规律，需临时开 Python

P3

字段含义对团队不透明

新成员和 Reviewer 看到魔数无法理解

P4

调试无中间过程可见

只能看最终结果，不知哪级管道出错

P5

边缘场景无批量验证

靠人工抽查，覆盖率无法保证

P6

测试用例无处留存

每次验证从零开始，知识无法积累

→
洞察汇聚
I1

「翻译」比「填写」更高频

读懂存量配置的需求远大于填写新配置。解码可见性优先于编码便利性。

I2

过程可见是工程调试的核心

工程师需要看「每一步发生了什么」，结果正确不代表过程正确。

I3

开发与验证是两种不同意图

不能共用界面，需明确切换，状态独立、可持久化。

→
设计方向
S1

表单替代位运算 + 水印常驻解码

双向翻译：填写时实时合成，阅读时水印解码。解码入口升为一级区块。

→ 03 详细设计
S2

管道逐级可视化

每个 |> 阶段单独展示输入、输出和错误信息，类比调试器单步执行。

→ 03 详细设计
S3

调试/验证双模式 + 用例持久化

两种模式状态独立，切换不丢失数据。失败行一键跳转调试区。

→ 03 详细设计

联调与迭代
前端接入 Demo 代码，设计侧根据联调反馈快速迭代，必要时生成调整版本。

02 概要设计

三个 Skill 概念分工

Step 02 · 读代码阶段

/code-to-product-doc

以代码仓为输入，输出面向体验设计师可读的产品文档。核心是「跨越工程语言和设计语言的壁垒」。

输入
代码仓目录 / 数据模型 / 样本数据
核心步骤
技术栈扫描模块识别确认数据模型分析逐模块深析
输出
product-doc.md可视化分析报告
内置「四步法」解释技术概念：是什么 → 为什么 → 怎么工作 → 对设计意味着什么
Step 03 · 设计分析阶段

/backend-to-design

以代码 + 产品文档为输入，输出设计机会点清单和工程级前端代码。是从「理解产品」到「改进产品」的跨越。

输入
代码仓 + product-doc.md
核心步骤
WCAG 2.1 AA 审查机会点挖掘工程代码生成
输出
设计机会点清单工程级前端代码
代码约束：纯 HTML+CSS+JS，无构建工具，与产品 Webview 技术栈完全对齐
Step 05 · 迭代交付阶段

/hifi-to-demo

以设计稿为输入，两阶段输出：先快速可点击原型供确认，再输出工程级组件代码。避免方向错误时的大量返工。

输入
设计稿截图 / Figma 链接 / 描述
Phase 1
单文件 HTML Demo（1 小时内）
Phase 2
Vue SFC / React TSX 组件Props 接口 + 基础测试
两阶段之间有明确「确认检查点」，方向错误只浪费 Phase 1 的时间
触发方式：Skill 安装在 ~/.claude/skills/ 目录下，在 Claude Code 里输入 / 触发，也支持自然语言描述自动匹配。
