项目背景 & 研究范围

研究范围 — 三个工具视图

SMC 偏移量计算器 批量表达式计算器 能效调速配置模板

openUBMC Studio 是运行在 VS Code Webview 内的固件工程师工具集。三个工具视图均由后端工程师先完成基础实现，是最高频使用的功能模块。

起点条件 — 后端先行，无产品文档

  • 后端工程师先完成工具视图基础实现，交付可运行 HTML Demo + 数据文件（YAML/JSON）

  • 没有产品经理同步出需求文档，没有设计规范参考
  • 技术概念密集：十六进制命令字、管道表达式、I²C 拓扑、CSR 对象层级
研究挑战：设计师在理解产品之前，需要先跨越「读懂工程代码」这道门槛。

01 研究洞察

目标用户 & 场景研究

场景 A · 开发阶段 — 构造命令字

工程师查到芯片手册：Fn=0x06、Cmd=0x4480、RW=1，需要手算位移合成 32 位整数，填入 SR 文件的 offset 字段。手算过程约 5–10 分钟，且错误只在运行时才暴露——命令字合法但字段写错，固件不报编译错误，硬件访问异常后才能发现。

高频 · 每天错误延迟暴露
场景 B · 调试阶段 — 解读日志数字

线上报错，日志显示命令失败，偏移量是 809893888（十进制）。工程师需要临时开 Python 做「十进制→十六进制→拆字段」三步转换，才能知道这是哪个功能模块在哪个设备上的哪类操作出了问题。每次调试都需要搭脚手架，中断工作流。

按事件触发每次搭脚手架
场景 C · 代码 Review — 理解魔数

同事代码里有 smc_read(0x30280001)，Reviewer 完全不知含义，无法判断字段值是否合理。需询问原作者或查阅两份对照文档。每次 Review 产生大量隐形沟通成本，且严重依赖原始开发者仍在团队。

低频隐形沟通成本

目标用户角色

角色主要操作频率
固件工程师编码（开发期）每天
调试工程师解码（排查期）按事件
代码 Reviewer解码（审查）低频
新团队成员两个方向学习期

操作安全性要求

  • 字段填错一位可能命令发给错误设备，校验必须即时且可读
  • 超出 32 位时代码取低 32 位（>>>0）— UI 必须显示溢出警告
  • offset 为变量引用（${Var}）时计算器需优雅处理
  • 管道表达式最长 1000 字符硬限制，超长不解析需明确提示
工程工具类产品：错误代价远高于消费类产品，安全性约束是设计的第一优先级。

技术概念解释 — 四步法

凡是「刚入职的体验设计师看到会有疑问」的技术概念，都用四步法解释：

① 是什么
用一句话，不用行话，「这是一种……用来……的东西」
② 为什么
工程师这样设计的动机——节省空间 / 提高效率 / 硬件规范
③ 怎么工作
用一个真实的值完整走一遍，必要时绘制结构图
④ 设计含义
翻译回界面语言：用户在哪一步接触它，UI 应该怎么处理

模块边界确认 — 内置检查点

Step 1.5 是整个 Skill 的关键控制点：AI 先列出识别到的模块清单，等设计师确认后再进入逐模块深析。

案例：SR 文件预览和 MIB 检查入口相邻，但是完全独立的功能单元——AI 无法自动判断，需设计师确认模块边界。
03 详细设计 · 案例

四步法应用 — SMC 命令字 0x30440100

设计师拿到代码时的状态

// 工程师的原始代码
cmd_word: 0x30440100

// ❌ 不知道这个数有几个字段
// ❌ 不知道每段代表什么含义
// ❌ 不知道 UI 应该让用户填什么
①

是什么

32 位整数，里面打包了 5 段信息——就像压缩文件，外面一个数，解压出来是多个独立字段。

②

为什么

BMC 硬件寄存器地址空间有限，多字段打包成一个整数传输效率更高，也与硬件规范对齐。

③

怎么工作 — 0x30440100 完整解析

Fn · 0x0C
Bit 31–26 · 6bit
功能码
温度管理
Cmd · 0x1100
Bit 25–10 · 16bit
命令编号
查芯片手册
MS
Bit 9
0=多
RW
Bit 8
1=读
Param · 0x00
Bit 7–0 · 8bit
附加参数
无附加

编码公式：(Fn<<26)|(Cmd<<10)|(MS<<9)|(RW<<8)|Param　关键认知：十六进制字符边界 ≠ 字段边界，无法直接读出各字段值，必须做位移运算。

④

对设计意味着什么

之前：一个整数输入框，用户手动填 0x30440100，需自己查两份对照表拼装
机会：5 个结构化字段选择框；底部实时预览拼装后的命令字；水印常驻解码
痛点核心：翻译比填写更高频——工程师花在「查找对照关系」上的时间远多于「填写命令」本身。

引入 Claude Design — 场景与摩擦

引入原因

Claude Code 在生成代码的交互逻辑方面能力强，但对视觉细节（字体精度、间距一致性、视觉层次）的优化精度偏低。引入 Claude Design 补足这一环节。

效果较好的使用场景

适用
已有基础 HTML（Code Demo 产出） 已有产品文档（/code-to-product-doc 产出）

以产品文档作为上下文输入 Claude Design，结合基础 HTML 做视觉优化——这是目前最顺畅的衔接方式。

当前工作流：product-doc.md → 输入 Claude Design → 视觉优化 HTML → 回到 Claude Code 做逻辑接入

主要摩擦点

无法直连
Claude Design 和 Claude Code 是两个独立工具，需手动跳转切换
上下文断裂
Code 中的分析上下文不能自动传递给 Design，需手动整理输入
格式不匹配
Code 生成的文件结构有时与 Design 的输入格式不完全对应

Design 介入前后对比

Claude Code 直接输出经 Claude Design 优化
交互逻辑✓ 完整✓ 完整
视觉细节精度⚠ 偏低✓ 提升明显
与高保真偏差较大缩小，但仍存在
切换成本—需手动跳转 + 重新整理上下文

05 体验评估

主要摩擦点发现

①

高保真交付仍是不可省的环节

  • Code Demo 已达交互和视觉「可用」，但与现网高保真的细节还原仍有明显距离
  • 字体精度、间距系统、边界状态处理等需要高保真补全
  • 当前结论：两份产出（Code Demo + 高保真）都需要交付给前后端
  • 设计侧工作量增加，两份产出之间的一致性需手动维护
探索中：Claude Design 在视觉优化环节有改善，但还不能完全替代高保真的细节精度。
②

Design 与 Code 的协同割裂

  • 两个工具独立，不能在 Claude Code 里直接唤起 Claude Design
  • 上下文不自动传递——Code 里的分析结果需手动整理后输入 Design
  • 文件格式有时不匹配，需要额外整理步骤
  • 当前绕过方式：产品文档作为中间载体手动传递
期望状态：在 Claude Code 里完成分析后，可直接「发送到 Claude Design」并携带上下文，无需重新整理。
③

模块边界判断需要人工确认

  • Step 1.5（功能模块识别）中，AI 有时难以准确区分「独立功能模块」和「同一模块的子功能」
  • 案例：SR 文件预览和 MIB 检查入口相邻，但是完全独立的功能单元
  • 需要设计师确认模块边界，再进入逐模块分析
  • 如果边界判断错误，后续所有分析的颗粒度都会偏差
Skill 里已内置「模块边界确认」步骤——识别后先向用户列出模块清单，确认后再继续，避免后续返工。
