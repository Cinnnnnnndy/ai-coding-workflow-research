从高保真设计稿到工程级 Code Demo

背景

  • 某批量操作界面已完成完整高保真设计稿
  • 需要快速生成可运行的 Code Demo 供前端联调参考
  • 目标：缩短"设计交付→代码实现"的沟通成本，减少理解歧义
路径 A 的出发点是已有设计稿，不需要经历代码解读与设计分析阶段

涉及步骤

Step 05

产出目标

  • Phase 1：HTML/CSS 静态 Demo（<1h）
  • Phase 2：Vue/React 工程组件（可迭代）
  • 同时保留高保真稿作为设计细节标准

Step 05 · 例子 A

联调验证 · 协同日志（2026-05-19）

设计侧观察 Code Demo 已达到「可用」状态——交互逻辑和视觉结构基本正确，主流程可以走通。
发现问题 与高保真稿存在细节精度差距：字体规格、内间距比例、边界状态（hover / disabled / error）的处理缺失。
协同决策 决定同时交付 Code Demo + 高保真设计稿——Code Demo 传达逻辑结构，高保真传达视觉细节。两份产物互补。
待优化 Claude Design 功能已接入，但在 Code 模式与 Design 模式之间切换的体验割裂，需要进一步探索协同方式。

关键洞察

  • Code Demo 不替代高保真稿，两者定位不同
  • 逻辑结构靠 Code Demo 传达，视觉细节靠高保真稿
  • /hifi-to-demo 显著提升从「有设计稿」到「有可运行代码」的速度
  • 边界状态仍需设计师手动补充到 Skill 输出中
后续方向：在 /hifi-to-demo 的 Skill 中补充边界状态的处理指引，减少联调摩擦
沉淀的 Skill · 例子 A

/hifi-to-demo

/hifi-to-demo
将高保真设计截图转化为可运行的工程代码。两阶段机制：Phase 1 快速生成 HTML Demo（<1小时），Phase 2 输出 Vue/React 工程级组件。
输入 高保真设计稿截图 + 交互说明（可选） Phase 1 静态 HTML + CSS Demo，浏览器可直接运行 Phase 2 Vue / React 组件，可接入工程，支持 Props / Emit 适用场景 已有高保真稿，需快速产出代码供联调或演示
安装方式：将 SKILL.md 放入 ~/.claude/skills/hifi-to-demo/，在 Claude Code 中输入 /hifi-to-demo 触发

适用判断

  • 有完整高保真截图或设计稿导出图
  • 需要快速给前端一个"可运行参考"
  • 想在联调前先对齐视觉和逻辑结构

不适用场景

  • 只有线框图或草图（精度不够）
  • 需要接入复杂业务逻辑或状态管理
  • 需要完整组件测试覆盖

例子 B · 后端代码→设计 · Step 01

批量表达式计算器 — 接收后端 Demo

背景

  • 后端工程师提供了批量表达式计算工具的原型 Demo
  • 工具用于对 BMC 传感器数据批量执行表达式计算
  • 核心挑战：管道语法表达式——非标准写法，设计师无法直接理解
  • 不理解功能就无法开始设计分析，Step 01 的任务是"搞清楚要设计什么"

管道语法示例

<=/Scanner_Lm75_Inlet.Value 当前值 ≤ 阈值
!=/Fan_PWM.Speed 转速 ≠ 设定值
>=/Thermal_Zone.Temperature 温度 ≥ 阈值

结构：[操作符]/[模块]_[组件].[属性]

Step 01 产出

接收后端 Demo，初步了解功能范围：

  • 明确"我们要设计什么"
  • 识别技术特殊性（管道语法）
  • 确认需要 /code-to-product-doc 介入
痛点：管道语法对设计师完全不透明。不经过翻译，无法识别用户操作路径，也无法开始设计分析。

下一步

运行 /code-to-product-doc，将后端代码转化为产品视角文档。

产出示例：批量表达式计算器

维度 产品视角描述
功能定义 批量对 BMC 传感器数据执行表达式计算，判断是否满足条件
核心操作 选择传感器 → 配置表达式 → 执行 → 查看结果

Step 03 · 例子 B

/backend-to-design — 痛点 → 洞察 → 策略

P · 痛点
P1

管道语法陌生

用户不熟悉 <= / 路径三段式格式，手动填写时出错率高

P2

调试与用例意图混乱

用户不清楚当前在「临时调试」还是「配置正式用例」，两者共用同一界面

P3

无历史记录

已配置过的表达式无法复用，每次从头输入，效率低

›
L · 洞察
L1

翻译比填写更高频

用户主要在「读懂」表达式含义，而非手动拼写——输入方式应转为引导选择

L2

操作意图决定界面结构

调试模式和正式配置的心智模型不同，需要在界面层明确区分

L3

历史 = 资产

已配置的表达式对用户有持续价值，复用需求高于新建需求

›
S · 策略
S1

语法引导 + 路径补全

路径选择器替代手动输入，操作符下拉选择 前端可独立实现

S2

调试 / 用例分离

独立的调试模式 vs 正式配置入口，状态区分 需后端支持

S3

历史面板 + 复用

历史表达式可收藏、一键复用 前端可独立实现

例子 B 产生的两个 Skill

/code-to-product-doc
将后端代码或工程 Demo 转化为产品视角文档。帮助设计师在没有产品文档的情况下快速理解功能，是进入设计分析的前置步骤。
输入 后端代码文件 / Demo 截图 / API 说明 输出 功能定义 + 操作路径 + 关键痛点识别 触发时机 收到后端 Demo，需要先理解「要设计什么」时 沉淀于 例子 B · Step 02
/backend-to-design
从产品文档或功能描述中，系统化提炼设计痛点、用户洞察与设计策略。输出 P-L-S 三列框架，区分前端可独立实现与需后端支持的策略。
输入 产品文档 / 功能描述 + 用户场景 输出 P 痛点 → L 洞察 → S 策略（含实现依赖标注） 触发时机 已有产品文档，需要提炼设计挑战与方向时 沉淀于 例子 B · Step 03
两个 Skill 构成路径 B 的核心工具链：先用 /code-to-product-doc 读懂代码，再用 /backend-to-design 转化为设计策略
