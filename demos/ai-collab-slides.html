/code-to-product-doc  后端代码 → 产品文档

一句话：把工程代码翻译成设计师能直接用的语言。这是后端先行流程的第一步，也是后续一切设计决策的工程锚点。

3 个典型触发场景（设计师视角）

  • 收到后端 Demo，看不懂功能。不理解就动不了手，要先把代码读成"在做什么"
  • 有 API 文档但缺产品视角。字段名是工程命名，需翻译成业务语言与 UI 显示对照
  • 要做设计审查、A11y 评估，但缺上下文。得先有产品定义才能比对什么算「好」

/backend-to-design  后端代码 → 设计策略 + 工程级实现

一句话：从后端交付物反向出设计审查 + 可访问性报告 + 机会点 + 工程级代码。连接「读懂代码」和「生成代码」的中间层。

3 个典型触发场景

  • 已有产品文档，需要做系统化设计审查。一致性、层级、可读性、A11y——靠经验逐项排查太慢
  • 要从代码反推设计规范。没有现成的 design system，需要从工程实现反向沉淀
  • 要给前端可直接接入工程的代码。不是 Demo，而是字段名对齐后端、含完整状态处理的可交付代码

P 痛点 → L 洞察 → S 策略 · 三列分析框架

P · Pain

痛点

管道语法不可读

非标准写法，新人/产品/QA 全部看不懂，错误定位靠猜

配置过程缺反馈

写一长串表达式后才能校验，错误一个个找

无 A11y 处理

配置面板无键盘可访问、无错误公告

L · Learning

洞察

表达式 = 三段式结构

operator / module / property，本身可视化即可降门槛

配置错误 = 引用错位

大多数错误来自引用了不存在的传感器或属性

A11y 是工程实现细节

aria-live、role、键盘焦点都有标准方案

S · Strategy

策略

FE 独立可做

表达式可视化拼接

用三列下拉选择 operator/module/property，避免手写

实时校验 + 错误定位

每段失焦即校验，错误高亮定位到具体段

完整 A11y 实现

aria-live 错误公告、键盘焦点管理、role 标注

同时产出 frontend/ 工程级代码：字段名对齐后端 data-model.yaml，包含 loading/empty/error 三态处理，前端可直接接入开发工程而不只是 Demo

/hifi-to-demo  设计稿 → Demo → 工程代码

一句话：让设计稿能跑起来。把「有高保真」到「有可运行代码」的时间从数天压缩到 1 小时内。

3 个典型触发场景

  • 评审/汇报前要可点的原型。静态图片说不清交互，需要可点击演示
  • 联调前要给前端可运行参考。设计稿 + 描述不够，前端实现常理解偏差
  • 要试技术可行性。布局引擎、动画方案、性能边界等需要代码层验证

两阶段机制

Phase 1 · < 1 小时
零依赖 HTML Demo

单文件，浏览器双击运行；用于评审、原型对齐

Phase 2 · 工程化
Vue / React 组件

支持 Props/Emit，可接入真实数据，含 Mock=API 契约
